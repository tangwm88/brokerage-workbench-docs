# 投行客户服务模型 v2.0（客观分析版 + 动作标准模型）

> **版本**：v2.0  
> **日期**：2026-05-31  
> **定位**：客户服务模块 = 业务标准模型 + 动作标准模型  
> **原则**：客观事实 + 状态判断 + 数据支撑，不评分、不主观  
> **体系归属**：五大模型体系之第五模块（客户分析→业务分析→方案/产品创设→交叉验证→客户服务）

---

## 一、整体模型体系框架

```
┌─────────────────────────────────────────────────────────┐
│              【投行智能体五大模型体系】                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ① 客户分析模型（底座）                                  │
│     └─ 客户画像：身份证 / 账户 / 关系网络 / 社会分层       │
│        作用：所有模型的统一数据底座，一次建立，全局同步      │
│                                                         │
│  ② 业务分析标准模型                                      │
│     └─ 定增业务分析模型（5维度：规模/定价/时机/募资/合规）  │
│     └─ IPO业务分析模型（待补充）                          │
│     └─ 可转债业务分析模型（待补充）                         │
│        作用：判断"项目能不能做"，回答业务可行性问题          │
│                                                         │
│  ③ 方案/产品创设模型                                     │
│     └─ 定增方案设计模型（3维度：怎么发/多少钱/干什么）      │
│     └─ 融资方案设计模型（待补充）                          │
│        作用：业务可行后，设计具体执行方案                    │
│                                                         │
│  ④ 交叉验证模型                                          │
│     └─ 合规验证 / 数据验证 / 逻辑验证（3维度）              │
│        作用：独立验证关键产出，防止盲区                     │
│                                                         │
│  ⑤ 客户服务模型（本文档）                                  │
│     ├─ 业务标准模型：客户维系状态客观分析（5维度）          │
│     └─ 动作标准模型：各角色客户服务动作（按角色）           │
│        作用：项目完成后/日常经营中，持续服务客户、发现新需求   │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  【流动关系】                                             │
│  客户分析 → 触发 ②/③/④/⑤                                │
│  ②业务分析 → 项目完成 → 触发 ⑤客户服务                    │
│  ⑤客户服务 → 发现新需求 → 触发 ②业务分析（循环）           │
│  ③方案创设 → 触发 ④交叉验证 → 修正 ③                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**核心设计原则**：
1. **客观分析，不评分**：用客观事实、数据指标、状态标签（正常/异常/严重）代替主观评分（X/5分）
2. **标准模型 + 动作模型双轨**：业务标准模型回答"状态是什么"，动作标准模型回答"该做什么"
3. **五大模块闭环**：客户分析是底座，业务分析判断可行性，方案创设设计路径，交叉验证把关质量，客户服务持续经营

## 二、客户服务业务标准模型（客观分析版）

> **定位**：回答"客户现在跟我们关系怎么样？还能做什么业务？"
> **输出风格**：客观事实罗列 + 状态判断（正常/异常/严重） + 数据支撑 + 清单化呈现

### 2.1 分析维度（5个）

| # | 维度 | 客观分析什么 | 数据来源 | 状态判断 |
|---|------|-------------|---------|---------|
| 1 | **服务交付状态** | 上次项目实际交付质量、遗留问题清单、客户明确反馈 | 项目交割记录 / 问题台账 / 客户书面/口头反馈 | [正常] / [有遗留] / [待修复] |
| 2 | **需求清单** | 客户明确表达的需求 + 可推断的潜在需求，每项带触发条件和时间窗口 | 客户公告 / 行业动态 / 财务数据 / 竞品动作 | [明确] / [待确认] / [待培育] |
| 3 | **关系网络状态** | 决策链每一层的联系人、互动频次、关系性质（业务/私人） | CRM记录 / 拜访日志 / 会议纪要 | [全通] / [部分打通] / [薄弱] |
| 4 | **信任事实** | 复购次数、推荐行为、信息开放度等可量化事实 | 项目台账 / 引荐记录 / 信息获取记录 | [高] / [中] / [低]（基于事实阈值） |
| 5 | **风险信号** | 竞品动作、客户内部变化、服务中断迹象、外部环境变化 | 竞品动态 / 人事变动公告 / 互动记录 / 行业数据 | [无异常] / [有信号] / [需干预] |

---

### 2.2 维度详解（客观分析框架）

#### 维度1：服务交付状态

**分析目标**：上次项目做完后，实际交付质量如何？有没有未结事项？客户有没有明确表达过不满或认可？

**客观分析框架**：
```
【项目交割事实】
  ├─ 项目名称：XX定增 / XX IPO / XX可转债
  ├─ 计划完成时间：202X年XX月
  ├─ 实际完成时间：202X年XX月（偏差：提前XX天 / 延后XX天 / 按期）
  ├─ 交付物清单：
  │   ├─ [已完成] 募集说明书及全套申报文件
  │   ├─ [已完成] 发行承销及募集资金到位
  │   ├─ [已完成] 上市或股份登记
  │   └─ [待完成] 持续督导期内信息披露（截止XX日）
  └─ 客户验收方式：书面确认 / 口头确认 / 未正式确认

【遗留问题清单】（逐项列出，不遗漏）
  ├─ 问题1：XX（状态：已解决 / 处理中 / 待启动，责任人：XX，截止日：XX）
  ├─ 问题2：XX（状态：XX，责任人：XX，截止日：XX）
  └─ 问题数量合计：X项（已解决X / 处理中X / 待启动X）

【客户反馈记录】（引用原话或原文）
  ├─ 书面反馈：XX（日期：XX，来源：邮件/函件/会议纪要）
  ├─ 口头反馈：XX（日期：XX，记录人：XX）
  └─ 反馈性质：[正面] / [中性] / [负面] / [无记录]

【同业对比参考】
  ├─ 同类项目平均交割周期：XX天（我方：XX天）
  ├─ 同类项目遗留问题率：XX%（我方：XX%）
  └─ 客户对同业同类项目反馈（如有）：XX

【状态判断】
  ├─ [正常]：按期交付，无遗留问题，客户反馈正面或无负面
  ├─ [有遗留]：有1-2项遗留问题在处理中，客户反馈中性或有轻微不满
  └─ [待修复]：遗留问题超过2项，或客户明确表达不满，需立即修复
```

**输出示例**（客观事实，无评分）：
```
服务交付状态：[有遗留]

事实罗列：
1. 项目：XX科技2025年定增，计划完成2025-12，实际完成2025-12（按期）
2. 遗留问题1项：持续督导期内信息披露规范辅导（状态：处理中，责任人：XX，截止：2026-06-30）
3. 客户反馈：董秘口头表示"整体满意，督导期希望多主动提醒"（记录人：XX，日期：2026-01-15）
4. 无书面投诉记录，无负面反馈记录

判断依据：遗留问题1项（未超期），客户反馈中性偏正面，无投诉。
结论：[有遗留] — 需在6月30日前完成督导辅导，消除遗留。
```

#### 维度2：需求清单

**分析目标**：客户现在明确需要什么？基于客观信号可以推断出什么需求？每项需求的触发条件和预计时间窗口是什么？

**客观分析框架**：
```
【显性需求清单】（客户明确表达或公开宣布的）
  逐项列出，每项包含：
  ├─ 需求描述：XX
  ├─ 信息来源：客户口头告知 / 公司公告 / 董事会决议 / 公开信息
  ├─ 获知日期：202X-XX-XX
  ├─ 预计启动时间：202X年QX（具体或范围）
  ├─ 预估价值：XX万元（按同类项目平均收费或客户预算范围）
  ├─ 我方准备度：[就绪] / [需准备] / [需培育]（基于我方可提供的产品/服务）
  └─ 状态跟踪：□ 已跟进确认 □ 方案准备中 □ 待启动

【隐性需求推断】（基于客观信号的推断，不臆测）
  逐项列出，每项包含：
  ├─ 推断需求：XX（如：并购财务顾问）
  ├─ 推断依据：（必须是客观事实，如：）
  │   ├─ 行业信号：所在行业近3个月发生XX起并购，平均标的规模XX亿
  │   ├─ 客户信号：客户产能利用率85%（行业均值70%），有扩张产能公告
  │   ├─ 竞品信号：主要竞争对手XX公司近期完成并购
  │   └─ 财务信号：客户现金储备XX亿，有投资性支出增加趋势
  ├─ 置信度：[高]（多信号交叉验证） / [中]（单一强信号） / [低]（弱信号推测）
  ├─ 预计触发条件：XX（如：产能利用率达到90%或竞品被收购）
  ├─ 预计时间窗口：202X年QX（推断范围）
  └─ 我方行动：□ 已准备案例 □ 已安排拜访了解 □ 待观察

【需求时间线】
  ├─ 2026 Q2：XX需求（状态：XX）
  ├─ 2026 Q3：XX需求（状态：XX）
  ├─ 2026 Q4：XX需求（状态：XX）
  └─ 2027：XX需求（状态：XX）

【状态判断】
  ├─ [明确]：显性需求≥1项且已确认，客户明确表达合作意向
  ├─ [待确认]：有显性需求信号但尚未与客户确认，或需求信息来自间接渠道
  └─ [待培育]：显性需求为0，隐性需求推断置信度低，需主动挖掘
```

**输出示例**：
```
需求清单：[待确认]

显性需求（已确认）：
1. 公司债券发行 | 来源：董秘口头告知 | 获知：2026-03-15
   预计时间：2026年Q4 | 预估价值：300万 | 我方准备度：[就绪]
   状态：□ 已跟进确认（待本周拜访确认具体规模和时间）

隐性需求推断（信号支撑）：
1. 并购财务顾问 | 置信度：[中]
   ├─ 依据1：行业信号 — 半导体封测行业近6个月发生并购3起，平均规模8亿
   ├─ 依据2：客户信号 — 客户产能利用率85%（高于行业均值70%），2025年报提及"择机整合产业链"
   └─ 依据3：竞品信号 — 主要竞争对手通富微电近期收购XX公司
   预计触发：产能利用率达90%或出现合适标的 | 时间窗口：2026年Q4-2027年Q1
   我方行动：□ 已准备案例（XX并购案例） □ 已安排拜访了解（本周拜访时顺带了解）

需求时间线：
  Q2 2026：确认债券发行细节
  Q4 2026：债券发行启动（如确认）
  Q4 2026-Q1 2027：并购需求可能触发（持续观察）

判断依据：显性需求1项已识别但待客户确认，隐性需求1项有客观信号支撑。
结论：[待确认] — 本周拜访确认债券发行意向，同步了解并购动向。
```

#### 维度3：关系网络状态

**分析目标**：我们跟客户决策链每一层的实际接触状态是什么？互动频率是多少？关系性质是纯粹业务还是包含私人信任？

**客观分析框架**：
```
【决策链覆盖事实】
  按层级逐项列出每个联系人：

  ├─ 最高决策层
  │   ├─ 职位：董事长/CEO（姓名：XX）
  │   ├─ 认识状态：[已认识] / [听说过] / [不认识]
  │   ├─ 首次认识时间：202X-XX（如适用）
  │   ├─ 近6个月互动次数：X次（电话/拜访/会议/活动）
  │   ├─ 互动形式：正式会议 / 业务拜访 / 私人聚会 / 活动偶遇
  │   ├─ 关系性质：[业务往来] / [有私交] / [深度信任]
  │   ├─ 信息开放度：主动分享战略信息 / 有问有答 / 被动回应
  │   └─ 我方主动触达：近6个月X次
  │
  ├─ 执行决策层
  │   ├─ 职位：董秘/财务总监（姓名：XX）
  │   ├─ 认识状态：[已认识]
  │   ├─ 首次认识时间：202X-XX
  │   ├─ 近6个月互动次数：X次
  │   ├─ 互动形式：XX
  │   ├─ 关系性质：XX
  │   ├─ 信息开放度：XX
  │   └─ 我方主动触达：近6个月X次
  │
  └─ 经办操作层
      ├─ 职位：证券事务代表/经办人（姓名：XX）
      ├─ 认识状态：[已认识]
      ├─ 首次认识时间：202X-XX
      ├─ 近6个月互动次数：X次
      ├─ 互动形式：XX
      ├─ 关系性质：XX
      ├─ 信息开放度：XX
      └─ 我方主动触达：近6个月X次

【关系网络健康度】
  ├─ 覆盖层级数：X层（共3层）
  ├─ 每层联系人数量：决策层X人 / 执行层X人 / 经办层X人
  ├─ 近6个月总互动次数：X次
  ├─ 近6个月我方主动触达次数：X次
  ├─ 信息开放度评估：高（≥2层主动分享） / 中（1层主动分享） / 低（被动回应）
  └─ 关系脆弱点：（客观列出）
      ├─ XX层级无接触（如：决策层仅认识但6个月无互动）
      └─ XX层级人员即将变动（如：董秘公告离职）

【状态判断】
  ├─ [全通]：3层全有联系人，近6个月每层≥1次互动，≥1层有信息主动分享
  ├─ [部分打通]：2层有有效互动，或3层有联系但1层互动频次极低
  └─ [薄弱]：仅1层有联系，或近6个月总互动≤3次

   └─ 我方主动触达：8次

关系网络健康度：
  ├─ 覆盖层级：3层全通（董事长/董秘/经办人）
  ├─ 近6个月总互动：22次
  ├─ 我方主动触达：13次
  ├─ 信息开放度：中（执行层有问有答，决策层被动回应）
  └─ 脆弱点：决策层（董事长）近6个月0次我方主动触达，仅2次被动参与活动

判断依据：3层均有联系人，但决策层互动频次低且全为被动，无主动触达。
结论：[部分打通] — 决策层关系需加固，建议安排总经理级主动拜访。
```

#### 维度4：信任事实

**分析目标**：客户对我们信任到什么程度？用可量化的事实说话，不用"感觉"。

**客观分析框架**：
```
【复购事实】
  ├─ 近3年合作项目总数：X个
  ├─ 项目类型覆盖：定增 / IPO / 可转债 / 债券 / 并购 / 其他（逐项列出）
  ├─ 项目时间线：
  │   ├─ 2024年：XX项目（类型，金额，状态）
  │   ├─ 2025年：XX项目（类型，金额，状态）
  │   └─ 2026年：XX项目（类型，金额，状态）
  ├─ 合作连续性：[连续]（每年≥1个项目） / [间歇]（间隔1年以上） / [孤立]（仅1个项目）
  └─ 客户主动发起复购次数：X次（客户主动找我方 vs 我方主动推销）

【推荐事实】
  ├─ 向其他企业推荐我方次数：X次（被推荐企业名称、时间、结果）
  ├─ 引荐同行或产业链伙伴次数：X次
  ├─ 公开场合提及或背书次数：X次（场合、时间、内容摘要）
  └─ 客户主动提供我方竞争情报次数：X次（时间、内容）

【信息开放事实】
  ├─ 提前告知我方重大决策次数：X次（决策内容、提前时间、信息性质）
  ├─ 分享未公开经营信息次数：X次（信息类型、分享场景）
  ├─ 邀请我方参与战略讨论次数：X次（讨论主题、我方角色）
  └─ 信息敏感度分级：公开信息 / 一般内部信息 / 敏感战略信息

## 三、客户服务动作标准模型（按角色设计）

> **定位**：各角色面对"客户服务"这个场景，该调用什么模型、做什么动作。
> **原则**：不是执行清单（打电话、发邮件），而是**分析决策模型**——每个角色用模型来判断、分析、决策，模型输出指导该做什么动作。

### 3.1 角色体系

| 角色 | 本质定位 | 核心诉求 | 工作台 |
|------|---------|---------|--------|
| **客户经理**（业务团队） | 客户全周期负责人 | 客户维系得好不好？还有什么需求？怎么巩固关系？ | 业务团队AI工作台 |
| **组织者** | 客户经营统筹者 | 哪些客户需要重点关注？资源怎么分配？ | 业务支持AI工作台 |
| **产品支持** | 方案交付者 | 客户需求对应什么产品？方案怎么准备？ | 业务支持AI工作台 |
| **质控内核** | 质量把关者 | 客户服务过程中有没有合规风险？ | 业务支持AI工作台 |

### 3.2 客户经理动作标准模型

#### 模型A：客户状态评估模型

**本质目标**：定期给客户做"体检"，客观评估5个维度的状态，输出健康度报告。

**调用触发**：
- 项目完成后1个月内（重点：服务交付状态）
- 每季度末（全面5维度评估）
- 发现竞品动作或客户异常信号时（重点：风险信号）
- 客户经理主动发起客户复盘时

**输入**：客户名称 + 客户分析底座数据（关系网络、历史项目、互动记录）

**输出**：
```
客户服务业务标准模型输出报告（5维度客观分析）
├─ 客户健康度：[健康] / [亚健康] / [不健康]
├─ 需求清单（显性+隐性）
├─ 关系网络地图（决策链覆盖状态）
├─ 关键行动清单（按P0/P1/P2排序）
└─ 价值预估（近1年 + 近3年LTV）
```

**与业务标准模型的关系**：这是业务标准模型的**调用入口**——客户经理通过这个模型触发完整的5维度客观分析。

---

#### 模型B：客户跟进策略模型

**本质目标**：根据客户状态评估结果，判断下一步该做什么、怎么做、谁来做。

**调用触发**：客户状态评估模型输出后，或客户经理需要制定具体跟进计划时。

**输入**：客户状态评估结果（5维度状态 + 需求清单 + 风险信号）

**维度（3个）**：

| 维度 | 名称 | 讲人话 | 输出 |
|------|------|--------|------|
| 1 | **接触策略** | 下一条接触谁（决策层/执行层/经办层）、用什么形式（拜访/电话/活动/材料）、谈什么话题 | 接触计划表 |
| 2 | **话题设计** | 这次接触的核心话题是什么？准备什么材料？预期客户反应？怎么应对？ | 话题脚本 + 材料清单 + 预期应对 |
| 3 | **时机判断** | 什么时候接触最合适？客户当前状态是否适合推进？有没有需要等待的信号？ | 时机判断 + 等待条件 |

**输出示例**：
```

#### 模型C：需求挖掘模型

**本质目标**：从存量客户中挖掘增量需求，判断显性需求怎么确认、隐性需求怎么培育。

**调用触发**：客户状态评估显示"需求清单：[待确认]"或"[待培育]"时。

**输入**：客户需求清单（显性+隐性）+ 客户行业数据 + 竞品动态

**维度（3个）**：

| 维度 | 名称 | 讲人话 | 输出 |
|------|------|--------|------|
| 1 | **需求验证** | 怎么确认客户真的有这个需求？问谁？问什么？用什么方式验证？ | 验证方案 |
| 2 | **需求培育** | 隐性需求怎么从"推断"变成"确认"？需要哪些信号？怎么创造触发条件？ | 培育计划 |
| 3 | **转化路径** | 需求确认后，怎么快速转化为业务机会？需要准备什么？多久能落地？ | 转化路径 + 时间表 |

**输出示例**：
```

├─ 验证对象：董秘（执行决策层，掌握融资计划）
├─ 验证方式：正式拜访 + 书面方案预沟通

│   ├─ Q1：公司Q4是否有债券融资计划？规模大概是多少？

│   └─ Q4：决策流程走到哪一步了？还需要董事会/股东会吗？

│   ├─ 同类债券发行案例3个（同行业/同规模）
│   ├─ 当前债券市场利率环境分析
│   └─ 我方承销优势一页纸
└─ 验证成功标准：董秘明确回应"有计划"并告知大致规模和时间

├─ 培育信号：
│   ├─ 信号1：产能利用率（当前85%，触发阈值90%）
│   ├─ 信号2：行业并购密度（近6个月3起，触发阈值5起或出现合适标的）
│   └─ 信号3：客户战略表述（年报提及"择机整合"）
├─ 培育动作：
│   ├─ 每季度提供行业并购动态简报（主动输出价值）
│   ├─ 产能利用率接近90%时主动提醒并询问扩张计划
│   └─ 出现合适标的时第一时间推送并询问意向

├─ 路径：需求确认 → 方案初稿（3个工作日） → 内部立项（1周） → 正式方案提交（1周） → 客户决策（2-4周） → 签署协议
├─ 预计总周期：2-3个月
├─ 关键节点：
│   ├─ T+3天：提交债券发行方案初稿
│   ├─ T+1周：内部立项完成，组建项目团队
│   ├─ T+2周：正式方案提交客户

```

### 3.3 组织者动作标准模型

#### 模型E：客户经营分析模型

**本质目标**：从全局视角判断客户群体的健康度，哪些客户需要重点关注、资源怎么分配。

**调用触发**：月度/季度客户经营复盘、年度客户分层评审、资源调配决策时。

**输入**：全部客户的客户服务业务标准模型输出（汇总）

**维度（4个）**：

| 维度 | 名称 | 讲人话 | 输出 |
|------|------|--------|------|
| 1 | **客户分层** | 按客观指标把客户分成核心/重要/一般/休眠，分层标准是什么？每层的标准是什么？ | 分层标准 + 客户列表 |
| 2 | **健康度分布** | 多少客户健康？多少亚健康？多少不健康？问题集中在哪些维度？ | 健康度分布图 |
| 3 | **资源匹配** | 高价值客户是否投入了足够资源？低价值客户是否占用了过多资源？ | 资源匹配分析 |
| 4 | **风险预警** | 哪些客户有流失风险？竞品在挖哪些客户？需要哪些干预？ | 风险预警清单 |

**分层标准（客观阈值）**：
```
客户分层标准：

【核心客户】（需同时满足）：
├─ 近3年合作项目≥2个，且总价值≥1000万
├─ 显性需求≥1项且已确认，或隐性需求推断置信度[高]
├─ 关系网络状态[全通]或[部分打通]（至少2层有效）
├─ 信任事实[高]或[中]（有复购基础）
└─ 风险信号[无异常]或[有信号]（可干预）

【重要客户】（需同时满足）：
├─ 近3年合作项目≥1个，或显性/隐性需求明确
├─ 关系网络状态[部分打通]（至少2层）
├─ 信任事实[中]或[低]但有升级空间
└─ 风险信号[有信号]或[需干预]（需维护）

【一般客户】（满足以下之一）：
├─ 近3年合作项目1个，暂无明确后续需求
├─ 关系网络状态[薄弱]（仅1层有效）
└─ 信任事实[低]，但无负面信号

【休眠客户】（满足以下之一）：
├─ 近3年无合作项目，且无明确需求信号
├─ 风险信号[需干预]且干预后无改善
└─ 客户明确表示不再合作或已转向竞品
```

**输出示例**：
```

├─ 重要客户：X家（名单：XX、XX、XX）
├─ 一般客户：X家
└─ 休眠客户：X家

健康度分布：
├─ [健康]：X家（%）— 服务交付正常+需求明确+关系稳固+信任高+无风险
├─ [亚健康]：X家（%）— 基础稳固但有1-2个薄弱环节
├─ [不健康]：X家（%）— 多个维度异常，需立即干预
└─ 问题集中维度：需求未确认（X家）/ 决策层薄弱（X家）/ 竞品威胁（X家）

资源匹配：
├─ 高价值客户（核心+重要）投入资源：XX人月/季度
├─ 低价值客户（一般）投入资源：XX人月/季度
├─ 资源匹配度：[匹配] / [高价值投入不足] / [低价值占用过多]

```

### 3.5 质控内核动作标准模型

#### 模型I：客户服务合规审查模型

**本质目标**：客户服务过程中的动作是否合规？有没有利益冲突？信息使用是否规范？

**调用触发**：客户服务方案涉及敏感操作时、客户投诉或纠纷时、定期合规抽查时。

**输入**：客户服务动作记录 + 相关协议/合同 + 客户反馈

**维度（3个）**：

| 维度 | 名称 | 讲人话 | 输出 |
|------|------|--------|------|
| 1 | **利益冲突审查** | 客户服务过程中是否存在利益冲突？（如：同时服务客户及其竞争对手） | 冲突审查结果 |
| 2 | **信息使用合规** | 客户提供的未公开信息是否被合规使用？有没有泄露风险？ | 信息使用审查 |
| 3 | **服务边界审查** | 客户服务是否超出协议范围？有没有过度承诺？ | 服务边界审查 |

## 四、同步响应机制

```
事件：客户经理完成客户服务业务标准模型分析
    ↓
【同步触发】
    ├─ 客户经理：
    │   ├─ 获得5维度客观分析结果
    │   ├─ 调用客户跟进策略模型 → 制定接触计划
    │   ├─ 调用需求挖掘模型 → 制定验证/培育/转化方案
    │   └─ 调用关系维护模型 → 制定修补/升级/防御方案
    │
    ├─ 组织者：
    │   ├─ 参考5维度状态 → 调用客户经营分析模型 → 客户分层
    │   ├─ 参考风险信号 → 调用客户资源调配决策模型 → 资源分配
    │   └─ 审批高层拜访等关键动作
    │
    ├─ 产品支持：
    │   ├─ 参考需求清单 → 调用交叉销售匹配模型 → 产品匹配
    │   ├─ 显性需求确认后 → 调用产品方案准备模型 → 方案准备
    │   └─ 协调跨业务线资源
    │
    └─ 质控内核：
        ├─ 参考服务交付状态 → 审查遗留问题合规性
        └─ 客户服务动作异常时 → 调用合规审查模型
    ↓
【同步汇聚】
    ├─ 客户经理：获得组织者确认的客户分层 + 资源调配方案
    ├─ 组织者：获得客户经理的客户健康度报告 + 风险预警
    ├─ 产品支持：获得需求-产品匹配清单 + 方案准备计划
    └─ 质控内核：获得客户服务合规审查记录
    ↓
【实时反馈】统一视图中各角色看到各自任务和状态
```

## 六、一页纸总结

```
┌─────────────────────────────────────────┐
│ 【投行客户服务模型 v2.0】                   │
├─────────────────────────────────────────┤
│ 1. 体系定位                               │
│    五大模型体系之第五模块：                 │
│    客户分析 → 业务分析 → 方案创设 → 交叉验证 → 客户服务│
│    项目完成后触发，发现新需求后回流业务分析      │
│                                         │
│ 2. 业务标准模型（客观分析，不评分）          │
│    ├─ 服务交付状态：客观事实（按期/遗留/反馈）    │
│    ├─ 需求清单：显性（逐项列来源/时间/价值）+    │
│    │             隐性（逐项列依据/置信度/触发条件）  │
│    ├─ 关系网络状态：决策链每层联系人+互动频次+    │
│    │                 信息开放度（无评分，纯事实）    │
│    ├─ 信任事实：复购/推荐/信息开放（可量化事实）   │
│    │             按阈值判定[高]/[中]/[低]           │
│    └─ 风险信号：竞品/内部/服务/环境（逐项列强度）  │
│                                         │
│ 3. 动作标准模型（按角色）                    │
│    ├─ 客户经理：                           │
│    │   客户状态评估模型（调用入口）             │
│    │   客户跟进策略模型（接触/话题/时机）        │
│    │   需求挖掘模型（验证/培育/转化）            │
│    │   关系维护模型（修补/升级/防御）            │
│    ├─ 组织者：                             │
│    │   客户经营分析模型（分层/健康度/资源/风险）  │
│    │   客户资源调配决策模型（优先级/分配/动作）   │
│    ├─ 产品支持：                           │
│    │   交叉销售匹配模型（需求-产品匹配）          │
│    │   产品方案准备模型（设计/差异化/交付）        │
│    └─ 质控内核：                           │
│        客户服务合规审查模型（冲突/信息/边界）       │
│                                         │
│ 4. 输出风格                               │
│    客观事实罗列 + 状态判断（正常/异常/严重）      │
│    数据支撑 + 清单化 + 无主观评分               │
│                                         │
│ 5. 与定增模型的关系                         │
│    定增模型：项目能不能做（项目视角）             │
│    客户服务：客户维系得怎么样（关系视角）          │
│    客户服务发现新需求 → 触发定增模型 → 循环      │
│                                         │
│ 【核心原则】                               │
│ 客观事实说话，状态判断代替评分，动作模型驱动行动    │
│ 持续经营客户，不是一锤子买卖                     │
└─────────────────────────────────────────┘
```
