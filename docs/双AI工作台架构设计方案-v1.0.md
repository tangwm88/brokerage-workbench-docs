# 双AI工作台架构设计方案

> **版本**：v1.0  
> **日期**：2026-05-30  
> **依据**：5月18日会议纪要（三端角色+闭环）+ 投行定增业务模型设计实践  
> **目标**：以完整闭环为目标，建设员工AI工作台与业务支持AI工作台，实现组织扁平化与能力开放

---

## 一、架构总览：两个工作台的定位

### 1.1 核心关系

两个工作台不是两套系统，而是**同一智能体架构的两个视图**——前端视角与后端视角。

```
┌─────────────────────────────────────────────────────────────────┐
│                         统一底座（自动驾驶大模型）               │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐             │
│  │  统一语料    │ │  身份打通    │ │  关系网络    │             │
│  └─────────────┘ └─────────────┘ └─────────────┘             │
└──────────────────────────────────┬────────────────────────────┘
                                     │ 共享
┌────────────────────────────────────┴────────────────────────────┐
│                                                               │
│  ┌─────────────────────────────┐  ┌─────────────────────────┐
│  │     员工AI工作台              │  │    业务支持AI工作台      │
│  │   （前端业务团队视角）        │  │  （中后台支持中心视角）   │
│  │                             │  │                         │
│  │  我能做什么？               │  │  我该怎么执行？          │
│  │  ├─ 自助分析（业务模型）    │  │  ├─ 接收服务请求          │
│  │  ├─ 提交需求（服务请求）   │◄─┤  ├─ 分解动作              │
│  │  ├─ 追踪进度（任务看板）    │  │  ├─ 执行标准动作          │
│  │  └─ 确认结果（闭环验收）   │  │  └─ 反馈结果              │
│  │                             │  │                         │
│  │  角色：承揽/承做/顾问/经理 │  │  角色：组织者+超级数字员工│
│  │                             │  │                         │
│  └─────────────────────────────┘  └─────────────────────────┘
│                    ▲                           │
│                    │      服务请求/状态同步    │
│                    └───────────────────────────┘
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 1.2 一句话定位

| 工作台 | 定位 | 核心问题 | 使用者 |
|--------|------|---------|--------|
| **员工AI工作台** | 前端业务团队的**能力入口** | "这个客户/项目，我该怎么判断？需要什么支持？" | 投行承揽、承做、保代、财富顾问、机构客户经理、营业部等 |
| **业务支持AI工作台** | 中后台支持中心的**动作执行系统** | "来了什么需求？该谁做？按什么标准做？什么时候做完？" | 各业务支持中心组织者+超级数字员工 |

### 1.3 设计原则

1. **模型共享，视角不同**：同一套业务标准模型，员工端看到的是"自助分析工具"，支持端看到的是"动作输入条件"
2. **动作闭环，状态可见**：员工提交的需求，在支持端变成任务，执行状态实时回传员工端
3. **组织者居中**：业务支持端必须有组织者角色，负责任务分解、分配、协调
4. **三端不割裂**：员工端可以直连协同端（研究所/资管等），支持端也可以呼叫协同端
5. **人定维度，AI执行**：模型维度由业务人员定义，AI只负责分析和执行，关键节点必须人工确认

## 二、员工AI工作台设计

### 2.1 角色体系

员工端不按部门划分，按**业务动作角色**划分——同一人在不同场景扮演不同角色。

```

│           员工AI工作台角色体系           │
├─────────────────────────────────────────┤
│                                         │
│  【通用角色】（所有业务线共享）          │
│  ├─ 业务开拓者：找线索、做初筛、触客户   │
│  ├─ 业务执行者：做方案、推流程、管项目   │
│  ├─ 业务管理者：盯进度、控风险、做决策   │
│  └─ 客户经营者：维护关系、挖需求、做复购 │

│  ├─ 投行：承揽人、承做人、保代、质控    │
│  ├─ 资管：产品经理、投资经理、客户经理   │
│  ├─ 财富：理财顾问、资产配置师、运营     │
│  ├─ 债券：交易员、承销专员、研究员     │
│  └─ 机构：机构客户经理、PB服务员      │
```

### 2.3 可调用能力清单

员工端可调用的能力分三类：

#### A. 自助分析能力（业务标准模型，直接调用）

| 能力名称 | 对应模型 | 调用方式 | 输出 | 权限 |
|---------|---------|---------|------|------|
| 客户分析 | 客户分析模型 | 输入客户名称 | 钱紧/股性/关系三维报告 | 全员 |
| 业务分析 | 业务分析模型 | 输入客户+业务类型 | 规模/价格/时机/募资四维报告 | 业务人员 |
| 方案生成 | 方案生成模型 | 输入分析结果 | 方案框架（怎么发/多少钱/干什么） | 承做及以上 |
| 客户服务 | 客户服务模型 | 输入客户名称 | 后遗症/需求/关系三维报告 | 全员 |
| 准入筛查 | 准入模型 | 输入客户名称 | 硬伤/前科/能力三维判断 | 全员 |
| 市场搜索 | 统一语料搜索 | 自然语言提问 | 公告/研报/舆情摘要 | 全员 |
| 竞品对比 | 可比案例模型 | 输入项目特征 | 3-5个可比案例 | 业务人员 |

**调用示例**：
```
用户输入：分析一下XX科技的定增机会
系统识别：客户分析模型 + 业务分析模型
输出：
  ├─ 钱紧不紧：现金流紧张，有融资需求 ⭐⭐⭐
  ├─ 股性好不好：股价处于低位，流动性一般 ⭐⭐
  ├─ 关系深不深：去年做过可转债，关系中等 ⭐⭐
  ├─ 发行规模：拟融资5亿，占市值15%，规模偏大 ⚠️
  ├─ 价格区间：当前股价8折有吸引力，但需看窗口
  ├─ 时机判断：监管近期放松，但市场偏弱，建议观望
  └─ 建议：值得跟进，但建议先确认客户意向和募资用途

用户操作：【提交服务请求】→ 立项支持
```

### 2.4 展业相关动作标准模型（员工端专属）

| 动作模型 | 场景 | 步骤 | 输出 |
|---------|------|------|------|
| 客户触达 | 首次接触客户 | 1.查客户画像 → 2.找切入点 → 3.准备话术 → 4.记录反馈 | 触达计划 |
| 需求挖掘 | 客户有初步意向 | 1.问融资目的 → 2.问时间窗口 → 3.问决策链 → 4.确认痛点 | 需求清单 |

| 关系维护 | 项目间隔期 | 1.监控客户动态 → 2.找由头互动 → 3.提供价值 → 4.记录关系 | 维护计划 |

## 三、业务支持AI工作台设计

### 3.1 核心设计原则

**支持中心不管"人"，只管业务和动作。**

- 人员组织属于HR和团队长
- 支持中心只按动作标准执行
- 每个动作有明确输入、输出、时限、责任人
- 不能"委托尽调"——动作责任人必须自己完成

### 3.4 组织者智能体核心逻辑

```python
# 组织者智能体任务分解逻辑（伪代码）
def 接收需求(员工请求):
    # Step 1: 意图识别
    需求类型 = 识别意图(员工请求.描述)
    
    # Step 2: 任务分解（匹配动作标准模型）
    if 需求类型 == "立项支持":
        任务清单 = [
            {"动作": "立项材料准备", "执行人": "项目分析员", "时限": "2日", "输入": "客户画像"},
            {"动作": "准入筛查", "执行人": "合规审查员", "时限": "并行", "输入": "合规数据"}
        ]
    elif 需求类型 == "方案设计":
        任务清单 = [
            {"动作": "方案设计", "执行人": "方案设计员", "时限": "3日", "输入": "分析结果"},
            {"动作": "可比案例", "执行人": "项目分析员", "时限": "并行", "输入": "案例库"}
        ]
    
    # Step 3: 分配执行
    for 任务 in 任务清单:
        分配给员工(任务.执行人, 任务)
        更新状态(员工请求, "已分配")
    
    # Step 4: 追踪与反馈
    while 有未完成任务:
        检查进度()
        if 超时或异常:
            升级处理()
        同步状态到员工端()
    
    # Step 5: 结果交付
    汇总结果(任务清单)
    反馈员工端(完成报告)
```

## 四、双工作台闭环流程

### 4.1 完整闭环示例：定增项目从线索到发行

```
【阶段1：线索发现】
员工端（承揽人员）
  ├─ 工作台搜索："最近有融资需求的科技企业"
  ├─ 调用客户分析模型 → XX科技：钱紧⭐⭐⭐ 股性⭐⭐ 关系⭐
  ├─ 判断：值得跟进
  └─ 操作：保存线索 → 准备触达

【阶段2：客户触达】
员工端（承揽人员）
  ├─ 调用展业模型：客户触达
  ├─ 输入：XX科技画像
  ├─ 输出：触达计划（切入点：现金流紧张 + 上次可转债合作）
  ├─ 执行：拜访客户，确认融资意向和规模
  └─ 操作：记录反馈 → 确认需求

【阶段3：立项申请】
员工端（承做人员）
  ├─ 提交服务请求：立项支持
  │    客户：XX科技
  │    规模：5亿
  │    类型：定增
  │    紧急：普通
  │
  └─ 业务支持端接收
       ├─ 组织者识别：立项支持 → 分解动作
       ├─ 分配项目分析员：立项材料准备（2日）
       ├─ 分配合规审查员：准入筛查（并行）
       ├─ 执行中...（状态实时同步员工端）
       └─ 完成 → 反馈立项材料包

【阶段4：方案设计】
员工端（承做人员）
  ├─ 收到立项通过通知
  ├─ 提交服务请求：方案设计
  │    客户：XX科技
  │    规模：5亿
  │    初步想法：竞价定增，用于扩产
  │
  └─ 业务支持端接收
       ├─ 组织者分解：
       │    ├─ 方案设计员：定增方案设计（3日）
       │    └─ 项目分析员：可比案例（并行）
       ├─ 执行中...（员工端可看进度）
       ├─ 发现需要行业研究 → 呼叫协同端（研究所）
       ├─ 研究所返回：行业研究报告
       ├─ 方案设计员整合 → 完成方案
       └─ 反馈员工端：方案建议书

【阶段5：材料制作】
员工端（承做人员）
  ├─ 确认方案
  ├─ 提交服务请求：募集说明书
  └─ 业务支持端
       ├─ 分配材料制作员（5日初稿）
       ├─ 自动生成：财务章节/募投章节/风险章节
       ├─ 人工复核：保代审核 → 内核审核
       └─ 反馈终稿

【阶段6：发行执行】
员工端（保代）
  ├─ 提交服务请求：发行执行
  └─ 业务支持端
       ├─ 分配发行执行员
       ├─ 簿记建档 → 投资者画像 → 需求分析
       ├─ 定价辅助 → 配售优化
       └─ 反馈发行结果

【阶段7：持续督导】
（自动触发，无需员工端提交）
业务支持端
  ├─ 项目督导员按周期执行
  ├─ 调用客户服务模型
  ├─ 监控募投进展 + 股价表现 + 投资者反馈
  ├─ 生成督导报告
  └─ 同步员工端：项目健康度 + 客户关系建议

【阶段8：闭环验收】
员工端
  ├─ 查看任务看板：全部完成
  ├─ 确认验收
  ├─ 评价支持中心服务质量
  └─ 客户经营：调用客户服务模型，找下一个业务机会
```

## 五、模型共享机制

### 5.1 业务标准模型：两边共享，视角不同

| 模型 | 员工端视角 | 支持端视角 |
|------|-----------|-----------|
| 客户分析 | "自助分析工具"：输入客户名称 → 看三维报告 | "动作输入条件"：立项材料准备的输入之一 |
| 业务分析 | "自助判断工具"：输入项目信息 → 看可行性 | "方案设计的输入条件"：定价/时机判断依据 |
| 方案生成 | "草稿生成工具"：输入分析结果 → 出框架 | "方案设计员的执行参考"：按框架深化 |
| 客户服务 | "客户关系工具"：输入客户 → 看维护建议 | "督导员的动作输入"：投后服务依据 |
| 准入模型 | "红线筛查工具"：输入客户 → 看能不能接 | "合规审查员的必做动作"：立项前置条件 |

### 5.2 动作标准模型：主要在支持端，员工端可见进度

员工端不直接执行动作标准模型，但可以通过任务看板**实时查看动作执行状态**。

## 七、实施路径

### 7.1 第一阶段：工作台框架搭建（2-4周）

1. **搭建统一底座**：统一语料/身份打通/关系网络（戴豫升/刘陶负责）
2. **开发工作台框架**：员工端三栏 + 支持端四栏（研发中心负责）
3. **定义通用协议**：服务请求格式/状态同步/协同呼叫（架构组负责）
4. **选择试点业务线**：建议投行或资管先跑（模型相对成熟）

### 7.2 第二阶段：试点业务线跑通（4-6周）

1. **填充模型**：按3-5维度标准，完成试点业务线的业务标准模型
2. **定义动作**：完成试点业务线的动作标准模型卡片
3. **配置角色**：试点业务线的组织者+超级数字员工
4. **跑通闭环**：选1-2个真实项目，从员工端提交到支持端完成，全链路验证

### 7.3 第三阶段：扩展与迭代（持续）

1. **扩展业务线**：按适配框架，逐个业务线上线
2. **优化模型**：每跑一个项目，收集反馈，迭代模型
3. **沉淀展业经验**：把优秀业务人员的展业方法沉淀为展业模型
4. **跨业务协同**：测试跨业务线呼叫（如投行呼叫研究所/资管）

## 八、关键成功因素

### 8.1 必须做对的事

| 序号 | 关键事项 | 失败后果 | 责任人 |
|------|---------|---------|--------|
| 1 | 模型维度压到3-5个 | 维度过多=没人用 | 业务负责人 |
| 2 | 语言讲人话 | 术语泛滥=前端拒绝 | 业务负责人 |
| 3 | 动作责任人不能转包 | 责任不清=质量失控 | 支持中心负责人 |
| 4 | 状态实时同步 | 员工不知道进度=反复催 | 技术负责人 |
| 5 | 关键节点人工确认 | 合规风险=监管处罚 | 合规负责人 |

### 8.2 常见陷阱

- **陷阱1：把支持中心做成"人工客服"** → 支持中心不是接电话的，是按标准动作执行的
- **陷阱2：模型做得太细** → 20个维度=没人看，3-5个维度才能指导行动
- **陷阱3：两个工作台割裂开发** → 必须统一架构、统一协议、统一底座
- **陷阱4：忽视展业模型** → 前端不仅需要分析工具，更需要"怎么展业"的指导
- **陷阱5：组织者智能体太弱** → 组织者是枢纽，意图识别不准=全流程崩

## 九、附录：术语表

| 术语 | 定义 |
|------|------|
| 员工AI工作台 | 前端业务团队使用的AI能力入口，支持自助分析、服务请求、协同呼叫 |
