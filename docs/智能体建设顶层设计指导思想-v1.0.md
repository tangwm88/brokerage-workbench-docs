## 一、核心论断：AI新世界

> **"AI新世界，99%的人都不行，就看谁转身快。"**  
> ——陶永泽总

### 1.1 旧世界已失效

旧知识、旧制度、旧规则、旧技术、旧数据大多已失效。  
**"没有行业，只有产业。"**万德行业分类、会计制度里的旧分类旧指标必须被质疑和重新定义。

### 1.2 模型即认知

- **模型即认知**：模型是业务标准与动作标准，是算法的基础
- **提示词即模型**：提示词工程就是模型工程，不能外包给技术团队
- **没有模型就没有算法**：不要将其神秘化

### 1.3 AI不淘汰人

AI不淘汰岗位，创造新岗位。  
放弃阈值设定等主观判断，将分析判断交给大模型，但人必须有情绪、有理想、有目的，有正确的事业心和价值观。

## 三、五大方法论要点

### 模型围绕业务闭环的五大类

**硬性标准**：模型围绕业务闭环主要包括五大类：

1. **客户分析** — 底座级，所有模型的基础
2. **业务分析** — 判断业务可行性
3. **方案设计** — 生成可执行方案
4. **交叉验证** — 独立验证，不是附加环节
5. **客户服务** — 闭环维护与复购

### 要点1：模型分两类

| 类型 | 定义 | 示例 |
|------|------|------|
| **产品/业务模型** | 有监管、有标准的业务模型 | 定增业务分析模型（发行规模/价格/时机/募资） |
| **服务模型** | 员工标准动作模型 | 客户触达动作、需求挖掘动作、方案推介动作 |

**关键**：两类模型必须分开设计，不能混为一谈。

### 要点2：客户分析是底座

**四个维度构建客户分析底座**：

1. **身份证** — 客户是谁
2. **账户** — 账户状态、资产状况
3. **关系网络** — 我方与对方关键人关系
4. **社会分层** — ⚠️ **不要叫"高净值"，按社会阶层分类**

**关键认知**：客户分析不是模型之一，是**所有模型的底座**。没有客户分析底座，上面的模型都是空中楼阁。

### 要点3：蒸馏同业做法

**方法**：
1. 找中金、摩根等对标机构
2. 用**大模型蒸馏**出标准动作
3. **人工校验取舍**，不是照搬

**关键**：不是看同业有什么，而是提取同业**怎么做**的标准动作。

### 要点4：交叉验证

**机制**：
- 一个智能体干活
- 一个智能体验证
- 关键输出必须双重确认

**关键**：交叉验证不是协同中的一个环节，是**独立的验证角色**。

### 要点5：反对旧思维

| 旧思维 | 新思维 |
|--------|--------|
| 讲协同 | 讲模型 |
| PPT展示 | 一页纸+一张图 |
| 主观画像 | 客观事实+因果逻辑 |
| 经验判断 | 大模型分析+人工定维度 |
| 追求完美再落地 | 快速迭代、先跑起来 |

**关键**：**"马上骂人，你别跟我讲这个，讲人话。"** ——所有输出必须质朴、准确、无歧义。

## 五、模型设计三原则（硬性标准）

### 原则1：透析本质，简化到3-5个维度

**硬性标准**：复杂问题必须简化到**最核心的3-5个维度**。

**案例**：
- ❌ 定增最初十几个甚至几十个维度 → 从业人员都难以驾驭
- ✅ 定增：**发行规模、价格、时机**（3个维度）
- ✅ 上市公司客户：**资产负债、现金流、资产配置**（3个维度）

**语言要求**：
- 彻底去除行业黑话
- 用质朴、准确、无歧义的语言描述
- **"马上骂人，你别跟我讲这个，讲人话"**

### 原则2：质疑旧制度，重新定义指标

**被点名批评的旧指标**：
- **研发费用**：多数软件企业将薪酬全部计入研发费用，未脱离"包工队"性质，没有价值
- **毛利率**：公式未包含销售费用，企业可轻易粉饰报表
- **旧会计制度**：大模型学到的多是"大路货"、"人云亦云"

**原则**：必须探究本质、重新定义，而非照搬旧制度。

### 原则3：信任大模型做分析，人只定维度

- 业务人员应**放弃阈值设定等主观判断**，将分析判断交给大模型
- **人只负责定义维度**
- AI不能取代人的思考——人必须有情绪、有理想、有目的

## 六、角色与闭环设计（三端模型）

### 三端角色

| 端 | 角色定义 | 职责边界 | 核心原则 |
|---|---------|---------|---------|
| **员工端** | 业务团队，人人可做业务 | 独立完成第一闭环 | 业务助理机器人，自助调用模型 |
| **业务支持端** | 各业务支持中心 | 不管"人"，只管业务和动作 | 人员组织属于HR和团队长 |
| **协同端** | 圆桌会议跨部门协同 | 支持中心→圆桌会议呼叫其他部门 | 被呼叫时才响应 |

### 三个闭环（平行可交叉，非线性串联）

```
闭环1：员工独立完成
  业务人员 → 调用客户分析底座 → 调用业务模型 → 自助判断 → 行动

闭环2：员工→支持中心
  业务人员 → 提交需求 → 超级智能体（组织者）→ 分解任务 → 超级数字员工执行 → 反馈

闭环3：支持中心→圆桌会议
  支持中心 → 呼叫协同端（研究所/资管/机构经纪/法务）→ 专业支持 → 返回
```

### 关键决策：委托尽调被叫停

> **项目责任人必须自己完成尽调，不能转包给系统或他人。**  
> 否则责任不清、效率与质量无法保证。

## 七、组织架构重构原则

### 7.1 围绕业务重构，不基于现有组织

- 不要基于华创证券现有组织框架做设计
- 围绕**客户、主要产品、主要业务**来构建
- 以客户服务为中心，以员工能力提升和收入增长为目的
- 旧的组织机构应被抛掉，组织方式随业务自然演变

### 7.2 业务支持中心=超级智能体

**核心定义**：
- 业务支持中心不是传统意义上的"部门"
- 是**超级智能体**，包含组织者+超级数字员工
- 通过员工工作台对外提供服务
- 不管"人"，只管业务和动作

**硬性标准**：
- **各业务支持中心归纳四个以内主要角色**
- 组织者角色是业务支持中心下面的角色之一，不是独立角色
- 示例：资管业务支持中心（组织者、投资经理、账户服务、产品设计）

## 十、输出标准：一页纸+一张图

### 10.1 禁止事项

- ❌ 长文本汇报（超过一页纸）
- ❌ PPT展示
- ❌ 主观画像
- ❌ 经验判断 without 数据支撑
- ❌ 术语和黑话
- ❌ 完美主义拖延

### 10.2 强制要求

- ✅ 一页纸讲清核心逻辑
- ✅ 一张图展示架构关系
- ✅ 3-5个维度，讲人话
- ✅ 客观事实+因果逻辑
- ✅ 快速迭代，先跑起来

## 十一、客户关系标准化三层

所有业务线的客户关系必须按此标准描述：

1. **是否为华创证券客户关系** — 是/否
2. **我方哪位员工与对方哪位关键人发生关系** — 董事长/总经理/财务总监/董秘等
3. **发生了什么事** — 具体业务事件

**禁止**：模糊描述如"关系良好"、"有联系"。  
**要求**：精确到人、到事、到时。

---

## 十二、客户分析底座标准

### 12.1 四个维度（硬性标准）

| 维度 | 定义 | 数据要求 | 禁止 |
|------|------|---------|------|
| **身份证** | 客户是谁 | 工商信息、实控人、股权结构 | 模糊描述 |
| **账户** | 账户状态、资产状况 | 账户数据、AUM、交易记录 | 叫"高净值" |
| **关系网络** | 我方与对方关键人关系 | 客户关系数据库、历史互动 | 主观评价 |
| **社会分层** | 客户在社会中的位置 | 行业地位、影响力、资源层级 | AUM分层 |

### 12.2 社会分层替代高净值

**关键升级**：传统的AUM分层/风险评级是旧世界的分类方式。  
AI时代要从**社会阶层视角**重新理解客户：
- 不是"有多少钱"
- 而是"在社会中处于什么位置"
- "掌握什么资源"
- "有什么影响力"

## 十三、执行检查清单

各项目组在提交方案前，必须逐项检查：

### 13.1 角色检查
- [ ] 是否明确目标角色？（只能有一个主要角色）
- [ ] 该角色的核心诉求是否一句话说清？
- [ ] 是否遵循"先定角色，再写模型"的顺序？

### 13.2 模型检查
- [ ] 维度数量是否控制在3-5个？
- [ ] 每个维度是否都是该角色"本质上"关心的？
- [ ] 维度之间是否有逻辑重叠？（如有，合并）
- [ ] 维度之间是否有逻辑矛盾？（如有，修正）
- [ ] 是否从角色视角出发？（而非系统/数据/技术视角）
- [ ] 语言是否"讲人话"？（目标角色能看懂）

### 13.3 动作检查
- [ ] 动作责任人是否明确？
- [ ] 是否违反"委托尽调"禁令？（责任人必须自己完成）
- [ ] 关键节点是否有人工确认机制？

### 13.4 输出检查
- [ ] 是否一页纸+一张图？
- [ ] 是否有交叉验证机制？
- [ ] 是否体现客户分析底座？
- [ ] 是否蒸馏了同业做法？

### 13.5 组织检查
- [ ] 是否基于现有组织框架？（如果是，打回重做）
- [ ] 是否围绕客户、产品、业务构建？
- [ ] 业务支持中心是否定义为超级智能体？

## 十四、常见错误与纠正

| 错误类型 | 错误示例 | 纠正方式 |
|---------|---------|---------|
| **角色混杂** | 一个模型同时服务"住客"和"服务人员" | 拆分为两个独立模型 |
| **维度过多** | 客户分析模型列出20个维度 | 硬性压缩到3-5个本质维度 |
| **系统视角** | 住客模型维度是"房态、渠道、库存" | 换成"期待、感知、评价" |
| **术语泛滥** | 用"RevPAR、ADR、OCC"描述住客体验 | 换成"性价比、舒适度、服务" |
| **主观画像** | "客户很有潜力"、"关系良好" | 换成客观事实+因果逻辑 |
| **旧指标照搬** | 直接用毛利率、研发费用等传统指标 | 质疑本质，重新定义 |
| **协同化思维** | "第一步、第二步、第三步..." | 模型化思维：维度+动作 |
| **追求完美** | "等所有条件成熟再上线" | 先跑起来，快速迭代 |
