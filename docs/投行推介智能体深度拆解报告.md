## 一、为什么选 Pitch Agent？

### 1.1 与我方战略目标的匹配度

| 我方战略目标 | Pitch Agent 对应设计 | 匹配度 |
|------------|---------------------|--------|
| **组织扁平化** — 中后台成为组织者，专业能力AI化后赋能前端 | MD（前端）提出一句话需求 → 研究员/建模师/文稿撰写（中后台AI代理）自动编排 → 输出完整推介材料 | ⭐⭐⭐⭐⭐ |
| **规定动作** — 事件-触发-步骤-输出-异常-时限 | 9步标准工作流，每步有明确触发条件、执行标准、输出物和审核关卡 | ⭐⭐⭐⭐⭐ |
| **角色体系** — 先有角色后有模型，维度服务于角色 | 研究员（只读）→ 建模师（计算）→ 文稿撰写（写入），三层角色各司其职 | ⭐⭐⭐⭐⭐ |
| **三句话法** — 可以/必须确认/不可以 | "禁止对外通讯"（不可以）+ "每个数字必须标注来源"（不可以）+ "Excel模型后和演示文稿后必须暂停提交审核"（必须确认） | ⭐⭐⭐⭐⭐ |
| **五步法** — 业务目标→标杆→数据→模型→验证 | 明确需求→行业概览→提取数据→展开可比→搭建并购→构建模型→生成足球场→填充演示文稿→质检 | ⭐⭐⭐⭐⭐ |
| **交叉验证** — 自洽、标杆、可执行、裁判挑刺 | ib-check-deck 质检技能 + 分步验证 + 敏感性分析自我校验 | ⭐⭐⭐⭐ |

### 2.2 九步工作流详解

```
Step 1: 明确需求
  触发：用户输入目标公司名称 + 一句话战略情况
  执行：确认目标、行业、情况；识别5-8个可比公司 + 5-10个先例交易
  输出：需求确认清单
  时限：即时
  
Step 2: 撰写情况概述
  触发：需求确认后自动执行
  执行：调用行业概览技能，起草公司快照和战略逻辑叙述
  输出：公司快照 + 战略逻辑段落
  技能：sector-overview
  
Step 3: 提取数据
  触发：情况概述完成后
  执行：调用数据接口获取交易倍数、先例交易数据、最新披露文件
  输出：原始数据块
  技能：数据接口（标普智汇、Daloopa）
  数据关联：明确需要哪些数据 → 从哪拿 → 怎么拿
  
Step 4: 展开可比公司
  触发：数据提取完成后
  执行：调用可比分析技能，以一致指标定义和异常值标记列出可比公司和先例交易
  输出：可比公司分析表 + 先例交易分析表
  技能：comps-analysis
  检查清单：数据源可追溯、公式引用单元格、统计量完整
  
Step 5: 搭建并购案例
  触发：可比分析完成后
  执行：调用杠杆收购模型技能，构建示意性杠杆收购
  输出：杠杆收购模型（进入/退出假设、资金来源与用途、回报敏感性）
  技能：lbo-model
  
Step 6: 构建模型其余部分
  触发：杠杆收购完成后
  执行：调用现金流折现模型和三表模型；遵循审计规范
  输出：现金流折现模型 + 三表财务模型
  技能：dcf-model, 3-statement-model, audit-xls
  关键规则：公式替代硬编码、分步验证、敏感性分析（奇数×奇数矩阵）
  
Step 7: 生成足球场图
  触发：所有模型完成后
  执行：汇总每种方法的最小/中位/最大值 — 可比、先例、现金流折现、杠杆收购
  输出：足球场估值汇总图
  关键规则：当前价格标记、方法间一致性校验
  
Step 8: 填充演示文稿
  触发：足球场图完成后
  执行：调用推介演示文稿技能，基于银行模板填充
  输出：推介演示文稿
  技能：pitch-deck, pptx-author, xlsx-author
  关键规则：每个幻灯片数字必须追溯至工作簿命名范围
  
Step 9: 演示文稿质检
  触发：演示文稿生成后
  执行：调用投行演示文稿质检技能
  输出：质检报告
  技能：ib-check-deck
  验证：合计数一致、脚注完整、日期统一、跨幻灯片数据一致
  
═══════════════════════════════════════════════════════
  🛑 质量关卡：Step 6（Excel模型完成后）必须暂停，提交银行家审核
  🛑 质量关卡：Step 9（演示文稿完成后）必须暂停，提交银行家审核
═══════════════════════════════════════════════════════
```

### 2.3 子代理安全分层

这是 Pitch Agent 最核心的设计之一，直接映射到我方"三句话法"中的"必须确认"和"不可以"原则：

| 层级 | 名称 | 接触外部数据？ | 权限 | 职责 | 与我方三句话法映射 |
|------|------|--------------|------|------|------------------|
| **Tier 1: 研究员** | pitch-researcher | ✅ 是（数据接口） | **只读**（Read, Grep） | 提取可比公司和先例交易数据，返回结构化JSON | 对应"可以"——内部数据分析，AI全自动 |
| **Tier 2: 建模师** | pitch-modeler | ✅ 是（数据接口） | **计算**（Read, Bash） | 基于输入数据运行现金流折现/杠杆收购计算，返回计算结果 | 对应"可以"——模型计算，AI全自动 |
| **Tier 3: 文稿撰写** | pitch-deck-writer | ❌ 否 | **写入**（Read, Write, Edit） | 唯一有权写入文件的代理，生成Excel和PowerPoint | 对应"必须确认"——输出物需人工审核 |
| **编排器** | pitch-agent（orchestrator） | ❌ 否 | **编排**（Read, Grep, Glob, Agent） | 任务分发、结果整合、质量关卡控制 | 对应"必须确认"——关键节点人工审核 |

**安全设计原则**：
- **写入层永远不接触外部数据源** — 避免不可信数据污染输出文件
- **计算层只读不写入** — 防止计算过程中的错误直接持久化
- **编排器不直接操作数据** — 只负责任务路由和代理调用
- **每个子代理有独立的模型上下文协议配置** — 最小权限原则

**关键洞察**：
- 引导事件支持**增量更新**（只刷新可比公司/足球场），而非每次都全量重建

- 每个事件都是**自然语言描述**，降低使用门槛

## 三、与我方五步法逐一对照映射

### Step 1: 业务目标（2日）

**我方标准**：
> 经营什么？关键动作？衡量指标？

**Pitch Agent 映射**：

| 我方问题 | Pitch Agent 回答 | 评价 |
|---------|-----------------|------|
| 经营什么？ | 为投行前端（MD/银行家）提供"一句话需求 → 完整推介材料"的能力 | 明确 |
| 关键动作？ | 9步标准工作流：明确需求→行业概览→提取数据→可比分析→杠杆收购→模型构建→足球场→演示文稿→质检 | 完整 |
| 衡量指标？ | **未明确** — 缺少动作完成率、超时率、人工介入率等量化指标 | ⚠️ 缺失 |

**差距与建议**：
- Pitch Agent 缺少明确的**衡量指标体系**，这正是我方五步法 Step 1 的核心产出
- 建议增加：
  - 推介材料首次通过率（%）
  - 从需求到交付的平均耗时（小时）
  - 人工审核轮次（平均）
  - 数据溯源覆盖率（%）
  - 跨幻灯片数据一致性错误率（%）

### Step 4: 模型定义（5日）

**我方标准**：
> 角色体系 + 分析模型（3-5维度） + 规定动作模型（事件-触发-步骤-输出-异常-时限）

**Pitch Agent 映射 — 角色体系**：

| 角色 | 职责 | 服务目的 | 被服务角色 | 执行角色 | 与我方对照 |
|------|------|---------|-----------|---------|-----------|
| **MD/银行家**（前端） | 提出需求、审核交付物 | 获得推介材料 | — | — | 服务接收方 |
| **研究员** | 提取可比公司和先例数据 | 为建模师提供输入 | 建模师 | AI代理 | 服务交付方 |
| **建模师** | 构建估值模型 | 为文稿撰写提供计算结果 | 文稿撰写 | AI代理 | 服务交付方 |
| **文稿撰写** | 生成Excel和PowerPoint | 为MD提供最终交付物 | MD | AI代理 | 服务交付方 |
| **编排器** | 任务分发、质量关卡 | 协调全流程 | 所有子代理 | AI代理 | 运营组织者 |

**评价**：角色定义清晰，无角色混乱 — **符合陶总第三大反对（反对角色混乱）**。

**Pitch Agent 映射 — 分析模型**：

| 分析维度 | 具体内容 | 与我方对照 |
|---------|---------|-----------|
| **可比公司分析** | 5-10家可比公司，经营指标+估值倍数+统计量 | 对应"企业客户分类模型"中的可比分析维度 |
| **先例交易分析** | 5-10个先例交易，交易倍数+溢价分析 | 对应"产品模型"中的交易结构维度 |
| **现金流折现估值** | 加权平均资本成本、终值、敏感性分析 | 对应"产品模型"中的估值方法论维度 |
| **杠杆收购分析** | 进入/退出假设、资金来源与用途、回报敏感性 | 对应"产品模型"中的交易结构维度 |
| **足球场汇总** | 多方法估值区间对比 | 对应"服务模型"中的交付标准维度 |

**评价**：5个分析维度，符合"每角色3-5个本质维度"的标准 — **符合陶总方法论**。

**Pitch Agent 映射 — 规定动作模型**：

| 动作 | 触发条件 | 执行步骤 | 输出物 | 异常处理 | 时限 | 验收标准 |
|------|---------|---------|--------|---------|------|---------|
| 可比分析 | 需求确认后自动触发 | 4步（标准化→匹配→分类→输出） | 可比公司表+先例交易表 | 数据源缺失→标记UNSOURCED | 60-90分钟 | 输出检查清单（8项） |
| 现金流折现建模 | 可比分析完成后自动触发 | 7步（数据→预测→自由现金流→加权平均资本成本→终值→现值→敏感性） | Excel模型 | 数据不一致→分步验证暂停 | 30分钟 | 零公式错误 |
| 推介演示文稿 | 所有模型完成后自动触发 | 5步（模板→映射→填充→格式化→验证） | PowerPoint文件 | 模板不匹配→上报用户 | 30分钟 | ib-check-deck质检通过 |

**评价**：每个动作都有完整的"事件-触发-步骤-输出-异常-时限"定义 — **与我方规定动作模型完全对齐**。

### Step 5: 交叉验证（3日）

**我方标准**：
> 自洽验证、标杆对比、可执行性模拟、独立裁判挑刺

**Pitch Agent 映射**：

| 验证类型 | Pitch Agent 做法 | 评价 |
|---------|-----------------|------|
| **自洽验证** | 敏感性分析中心单元格 = 基础案例实际值；跨幻灯片数据一致性检查 | ✅ 自动化 |
| **标杆对比** | 可比公司统计量（25/50/75分位）提供市场基准 | ✅ 内置 |
| **可执行性模拟** | 分步验证：每步完成后向用户展示并确认 | ✅ 内置 |
| **独立裁判挑刺** | **缺失** — 无独立第三方验证机制 | ⚠️ 缺失 |

**差距与建议**：
- Pitch Agent 的验证偏"技术性自洽"，缺少"独立裁判"环节
- 我方五步法要求的"独立裁判挑刺"可补位：引入独立合规代理或资深银行家评审

## 六、关键借鉴点（按优先级排序）

### 🔴 高优先级（本周可落地）

#### 借鉴点1：三层安全分层设计（只读→计算→写入）

**来源**：Pitch Agent 子代理配置

**我方移植方案**：
```
┌─────────────────────────────────────────────────────────────┐
│  我方投行智能体（拟建）                                        │
├─────────────────────────────────────────────────────────────┤
│  只读层：数据提取员                                            │
│    • 工具：读取、搜索                                          │
│    • 数据接口：Wind/同花顺 MCP（只读）                         │
│    • 职责：提取企业财务数据、可比公司数据、先例交易数据          │
│    • 输出：结构化JSON（schema校验）                            │
├─────────────────────────────────────────────────────────────┤
│  计算层：模型构建员                                            │
│    • 工具：读取、计算（Python/Bash）                           │
│    • 数据接口：无（仅使用只读层输出）                          │
│    • 职责：构建现金流折现/杠杆收购/三表模型                     │
│    • 输出：计算结果JSON                                        │
├─────────────────────────────────────────────────────────────┤
│  写入层：材料撰写员                                            │
│    • 工具：读取、写入、编辑                                    │
│    • 数据接口：无（不接触外部数据）                            │
│    • 职责：生成Excel工作簿和PowerPoint                         │
│    • 输出：./out/ 目录下的文件                                 │
└─────────────────────────────────────────────────────────────┘
```

**适配国内监管**：
- 只读层的数据接口从"标普智汇/Daloopa"替换为"Wind/同花顺/交易所披露"
- 写入层生成文件前需通过**合规审查代理**校验（三句话法合规）

#### 借鉴点2：分步验证机制（Step-by-Step Verification）

**来源**：现金流折现模型技能定义

**关键规则**：
```
• 数据提取后 → 展示原始输入块，确认后再预测
• 收入预测后 → 展示预测顶线和增速，确认后再构建利润率
• 自由现金流构建后 → 展示完整自由现金流表，确认后再计算加权平均资本成本
• 加权平均资本成本后 → 展示计算和输入，确认后再贴现
• 终值+现值后 → 展示权益桥（企业价值→权益价值→每股），确认后再做敏感性分析
• **禁止从头到尾一次性构建后再展示**
```

**我方移植价值**：
- 直接嵌入我方规定动作的"验收标准"环节
- 每个步骤的交付物都有明确的"展示→确认→继续"机制
- 降低错误累积风险，符合陶总"反对拍脑袋和甩锅给大模型"

#### 借鉴点3：反模式清单（Anti-Patterns）

**来源**：推介演示文稿技能定义

**三个核心反模式**：
1. **将数据填入占位框** — 保留彩色指示框而非替换为生产格式
2. **文本"表格"** — 用 | 字符创建假表格而非真正的表格对象
3. **继承占位对比度** — 保留浅色文字在彩色背景上的占位配色

**我方移植**：
- 债券募集书常见反模式：硬编码利率、未标注数据来源、格式不一致
- 适当性匹配常见反模式：混淆投资者类型、忽略风险等级、未记录匹配过程
- 客户分析常见反模式：以偏概全的分类、模糊的维度定义

### 🟡 中优先级（1个月内落地）

#### 借鉴点4：敏感性分析规范

**来源**：现金流折现模型技能定义

**关键规则**：
- 使用**奇数×奇数**矩阵（标准5×5，有时7×7）— 确保真正的中心单元格
- **中心单元格 = 基础案例** — 中心单元格的输出必须等于模型实际隐含股价
- **高亮中心单元格** — 中蓝色填充+粗体，一眼可见基础案例
- **全部单元格填充完整公式** — 禁止占位符、线性近似、手动步骤

**我方移植价值**：
- 债券定价敏感性分析、收益率曲线平移分析可直接套用
- 投后管理的净值波动敏感性分析可借鉴

#### 借鉴点5：数据源优先级强制规范

**来源**：可比分析技能定义

```
**必须遵循的数据源优先级：**
1. 首选：专用数据接口（Wind MCP、同花顺 MCP）
2. 数据接口可用时，禁止网络搜索
3. 仅当数据接口不可用时：交易所披露、证监会公告
4. 绝不以网络搜索作为主要数据源
```

**我方移植价值**：
- 明确的数据源优先级 = **可追溯 + 可审计** — 满足监管要求
- 每个硬编码输入必须有批注标注来源 — 满足合规审查

#### 借鉴点6：输出检查清单（Output Checklist）

**来源**：可比分析技能定义（8项）、推介演示文稿技能定义（15项）

**我方标准化建议**：
每个规定动作配一个输出检查清单（5-10项），嵌入动作模型的"验收标准"环节。

### 🟢 低优先级（参考思路）

#### 借鉴点7：引导事件（Steering Events）

**来源**：Pitch Agent 引导示例

支持增量更新（"仅刷新可比公司和足球场"）和不同场景（买方明确 vs. 卖方探索）。

**借鉴思路**：我方工作台可设计类似的"自然语言引导事件"，降低前端使用门槛。

**来源**：脚本工具/技能同步

通过 `sync-agent-skills.py` 确保智能体包中的技能与垂直技能源文件同步。

**借鉴思路**：开发我方版工具，确保各业务线的动作模型与统一参考方法论同步。

## 九、结论

**Pitch Agent 是 Anthropic 全仓库中与我方建设方法论最匹配的智能体。**

**核心匹配点**：
1. **组织扁平化的最佳实践** — "MD一句话需求 → AI自动编排中后台能力 → 输出完整交付物"的模式，与我方"业务支持中心成为组织者，专业能力AI化赋能前端"的战略完全对齐
2. **规定动作模型的完整范本** — 9步工作流、三层角色、技能依赖、质量关卡，每个环节都有明确的"事件-触发-步骤-输出-异常-时限"定义
3. **三句话法的自然落地** — "禁止对外通讯"（不可以）+ "标记无来源数据"（不可以）+ "关键节点暂停审核"（必须确认）
4. **五步法的高度对齐** — 从业务目标（明确需求）到交叉验证（ib-check-deck），每个步骤都有对应

**移植优先级**：
- **立即行动**：三层安全分层设计、分步验证机制、反模式清单
- **1个月内**：敏感性分析规范、数据源优先级、输出检查清单标准化
- **3个月内**：完整投行智能体（角色体系 + 技能体系 + 编排器 + 工作台集成）

**最终判断**：Pitch Agent 不仅是一个可借鉴的参考实现，更是我方建设投行智能体的** Blueprint（蓝图）** — 它的架构设计、角色分工、工作流编排、质量关卡都经过 Anthropic 的工程化打磨，我方只需在**数据源（Wind/同花顺替代标普智汇/Daloopa）、监管合规（三句话法嵌入）、语言环境（中文替代英文）**三个维度进行适配，即可快速落地。
