## 一、解决方案概述

### 1.1 核心目标

**不是给现有系统加AI功能，而是让AI重构组织运营方式。**

本方案解决的根本问题：证券公司传统组织架构中，专业能力被锁在部门墙后面，前端团队需要层层请示才能调用中后台能力，导致响应慢、损耗大、客户体验差。

**目标状态**：
- 业务支持中心（中后台）成为**组织者**——把专业能力提炼成可调用的模型和标准动作
- 前端团队（客户经理/投顾/投行前台）成为**能力调用者**——直接调用AI工作台完成业务
- 组织层级压缩，决策链路缩短，从"层层汇报"变为"直接调用"

### 1.2 什么是AI原生应用

**不是"传统软件+AI模块"，而是"AI定义业务逻辑，人做关键确认"。**

| 维度 | 传统软件思维 | AI原生思维 |
|------|------------|-----------|
| 核心驱动 | 按固定流程走 | 按客户需求动态调用模型 |
| 角色关系 | 人操作软件 | 智能体执行标准动作，人做关键判断 |
| 数据使用 | 记录和查询 | 预判驱动、主动推荐、自动关联 |
| 组织影响 | 维持原有层级 | 打破部门墙，能力直接开放 |
| 交付形态 | 功能模块 | 角色工作台 + 可调用模型库 |

## 二、组织重构：从层级组织到能力平台

### 2.1 问题诊断：传统组织为什么慢

```
传统模式（层级式）：

客户请求 → 客户经理 → 部门主管 → 分公司负责人 → 总部业务部门
              ↓           ↓            ↓              ↓
           请示方案    请示资源    协调多个部门    调配专业能力
              ↓           ↓            ↓              ↓
           等3天       等2天       等5天          等7天
              ↓           ↓            ↓              ↓
           回复客户 ← 汇总 ← 各部门分别回复 ← 专业意见

总耗时：2-3周，客户早就走了。
```

**核心问题**：
1. 专业能力被锁在部门墙后面，前端够不着
2. 每个请求都要层层请示，决策链路太长
3. 相同的问题不同部门重复解答，没有标准化沉淀
4. 中后台被动响应，不是主动组织赋能

### 2.2 目标架构：组织者 + 超级数字员工


### 2.3 组织重构的三个关键变化

**变化1：中后台从"被动响应"变为"主动组织"**

| 传统 | AI原生 |
|------|--------|
| 前端来问，中后台回答 | 中后台提前把能力提炼成模型，前端直接调用 |
| 每个请求重新想方案 | 标准模型自动匹配，特殊情况才升级人工 |
| 中后台做具体执行 | 中后台做模型设计、质量监督、异常升级 |

**变化2：前端从"层层请示"变为"直接调用"**

| 传统 | AI原生 |
|------|--------|
| 不确定怎么做 → 问主管 → 问部门 → 等回复 | 打开工作台 → 选择业务类型 → 输入客户信息 → 系统自动调用模型 |
| 靠个人经验判断 | 靠标准化模型判断，经验沉淀为模型 |
| 做一遍学一遍 | 系统记录所有动作，自动优化模型 |

**变化3：管理层从"听汇报"变为"看穿透"**

| 传统 | AI原生 |
|------|--------|
| 每周看报表 | 实时看管理驾驶舱 |
| 听各级层层汇报 | 穿透到具体客户、具体项目、具体员工动作 |
| 事后发现问题 | 实时预警异常（超时、违规、风险） |

### 2.4 组织扁平化的边界

**不是消灭所有层级，而是压缩决策链路。**

| 层级 | 保留/压缩 | 角色变化 |
|------|----------|---------|
| 公司领导层 | 保留 | 从"听汇报做决策"变为"看驾驶舱定战略" |
| 业务支持中心 | 核心组织者 | 从"做具体事"变为"设计模型+监督质量+处理异常" |
| 分公司/营业部 | 压缩 | 从"传声筒"变为"客户一线触达+简单调用" |
| 客户经理/投顾 | 保留 | 从"填表跑审批"变为"客户经营+关键判断确认" |
| 后台执行人员 | 大幅减少 | 标准化动作由AI执行，人做复杂判断和例外处理 |

## 三、三大模型体系

### 3.1 模型体系总览

```
┌──────────────────────────────────────────────────────────────┐
│                     三大模型体系                              │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌─────────────────┐                                         │
│  │    产品模型      │  ← 回答"我们有什么？标准是什么？"         │
│  │  (业务标准模型)  │                                         │
│  │                  │                                         │
│  │  产品分类库      │  产品维度、规格标准、定价标准、准入标准      │

│  └─────────────────┘                                         │
│                                                               │

│  │    服务模型      │  ← 回答"怎么交付？标准是什么？"            │

│  ┌─────────────────┐                                         │
│  │   分角色动作模型  │  ← 回答"谁做什么？怎么做？什么时候做完？"   │
│  │  (规定动作模型)  │                                         │
│  │                  │                                         │
│  │  角色定义        │  组织者1 + 超级数字员工3-6                  │
│  │  动作触发        │  什么事件启动 → 什么角色响应                 │
│  │  标准步骤        │  每一步做什么 → 输出什么 → 异常怎么办        │
│  │  时限约束        │  刚性deadline → 超时预警 → 升级机制          │
│  └─────────────────┘                                         │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

**三个模型不是孤立的，而是动态调用的组合。**

客户请求 → 组织者调用**客户分析模型**（服务模型子集）→ 匹配**产品模型** → 生成**动作指令**（分角色动作模型）→ 各角色执行 → 组织者汇总 → 交付客户

### 3.2 产品模型（业务标准模型）

**定义**：把企业的所有产品（服务、方案、工具）按客观维度分类描述，形成可比较、可验证、可被AI调用的标准库。

**不是**：产品说明书堆砌、营销话术汇总、功能列表罗列
**而是**：客观维度 + 可比较标准 + 可调用接口

#### 3.2.1 产品模型结构

```yaml
产品模型示例：债券融资产品库

产品大类: 公司债券
产品子类:
  - 产品名称: "公开发行公司债券"
    客观维度:
      发行条件:
        - 净资产要求: "≥3000万（有限责任）/≥3000万（股份）"
        - 累计债券余额上限: "≤净资产的40%"
        - 近3年平均可分配利润: "≥1年利息"
        - 资金用途: "核准用途，不得用于弥补亏损、非生产性支出"
      发行要素:
        - 发行期限: "3年/5年/7年/10年"
        - 发行规模: "1亿-100亿（视净资产而定）"
        - 票面利率确定方式: "簿记建档/询价"
        - 担保方式: "无担保/资产抵押/第三方担保/质押"
      适用客户画像:
        - 企业类型: "上市公司/非上市公众公司/非金融企业"
        - 主体评级: "AA及以上（市场惯例，非强制）"
        - 产业属性: "基建/制造/能源/消费等"
        - 资金需求特征: "中长期项目资金/补充流动资金/置换债务"
      准入标准:
        - 财务硬约束: "资产负债率≤70%，近3年无连续亏损"
        - 合规硬约束: "近3年无重大违法违规，无 suspended"
        - 评级软约束: "建议AA+以上，AA需增信"
      比较维度:
        - 与中期票据对比: "交易所 vs 银行间/投资者结构不同/流动性差异"
        - 与定向工具对比: "公开 vs 非公开/投资者范围/信息披露要求"
    可调用接口:
      输入: [企业名称, 主体评级, 净资产, 产业属性, 资金用途, 期望期限]
      输出: [产品匹配度评分, 适用产品列表, 关键约束提醒, 同案例参考]
      置信度: "基于公开数据自动计算，人工复核后确认"

  - 产品名称: "非公开发行公司债券"
    客观维度:
      发行条件:
        - 投资者范围: "合格投资者/特定投资者（≤200人）"
        - 净资产要求: "同公开发行"
        - 其他: "无公开市场交易要求"
      ...
```

#### 3.2.2 产品模型设计原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **客观描述** | 用可量化、可验证的维度，不用主观评价 | "优质企业" → 改为"净资产≥10亿，主体评级AA+" |
| **分类清晰** | 产品归产品，营销归营销，不混在一起 | 不要把"客户满意度"塞进产品模型，那是服务模型 |
| **可比较** | 同类产品之间能直接对比，差异一目了然 | 两个债券产品的差异用表格对比，不是文字描述 |
| **可调用** | AI能根据客户输入参数，自动匹配产品 | 不是让客户自己翻产品手册 |
| **动态更新** | 监管规则、市场惯例变化时自动更新 | 40%净资产上限规则变化，模型自动更新 |

#### 3.2.3 产品模型覆盖范围

按业务线分类建设产品模型库：

| 业务线 | 产品大类 | 产品子类数量（预估） | 优先级 |
|--------|---------|-------------------|--------|
| 投资银行（股权） | IPO、定增、并购、可转债 | 15-20 | P1 |
| 投资银行（债券） | 公司债、企业债、ABS、非金融企业债务融资工具 | 20-30 | P0 |
| 资产管理 | 公募基金、专户、资管计划、养老金产品 | 25-35 | P0 |
| 财富管理 | 现金管理、固收、权益、混合、另类 | 30-40 | P0 |
| 机构经纪 | 交易通道、托管、外包、研究服务 | 15-20 | P1 |
| 信用交易 | 融资融券、股票质押、约定购回 | 8-12 | P1 |
| 研究所 | 研究报告、路演、咨询、数据服务 | 10-15 | P2 |
| 固收投资 | 利率债、信用债、衍生品、货币基金 | 15-20 | P2 |

### 3.3 服务模型（服务交付模型）

**定义**：定义企业向客户交付服务的标准——从客户请求触发到服务完成的全链路标准，包括响应标准、完成标准、升级标准、监督标准。

**不是**：服务流程图、客服话术手册、SLA承诺数字
**而是**：触发条件 → 响应动作 → 完成标准 → 升级机制 → 监督闭环

#### 3.3.1 服务模型结构

```yaml
服务模型示例：客户开户服务

服务名称: "客户开户服务"
服务目标: "在合规前提下，让客户以最少步骤完成开户，当日可交易"

触发条件:
  - 客户提交开户申请（APP/网页/线下）
  - 客户身份已通过反洗钱初筛
  - 客户风险测评问卷已完成

响应标准:
  首次响应: "系统自动确认收到申请，2分钟内推送待办给客户经理"
  人工介入: "智能体预审不通过 → 标记原因 → 推送人工复核"
  并行处理: "身份验证、反洗钱核查、适当性匹配可同步进行"

完成标准:
  成功标准:
    - 客户资金账户开立完成
    - 交易权限开通（股票/基金/债券按客户选择）
    - 客户收到确认短信和APP推送
    - 客户可在当日交易时间内开始交易
  失败标准:
    - 身份验证不通过 → 明确告知客户原因和补充材料要求
    - 反洗钱命中 → 进入强化尽调流程，明确时限和联系人
    - 适当性不匹配 → 推荐合适的产品类别，原申请被拒绝并记录原因

升级标准:
  升级触发:
    - 客户身份存疑（系统置信度<80%）
    - 反洗钱命中高风险名单
    - 客户坚持要求超适当性权限的产品
    - 客户投诉开户体验
  升级路径:
    - 身份存疑 → 升级至合规审查员（2小时内响应）
    - 反洗钱高风险 → 升级至反洗钱专员+合规主管（4小时内响应）
    - 超适当性坚持 → 升级至投顾+合规（先沟通再决定，1工作日内）
    - 客户投诉 → 升级至客户关怀员+营业部负责人（30分钟内响应）
  升级时限: "每级升级必须在规定时间内响应，超时自动升级至上一级"

监督标准:
  质量检查点:
    - 开户资料完整性（系统自动扫描）
    - 客户身份真实性（与公安系统比对）
    - 适当性匹配准确性（与产品模型交叉验证）
    - 客户体验（开户完成后推送满意度问卷）
  时限检查:
    - 首次响应 ≤ 2分钟
    - 普通开户完成 ≤ 1工作日
    - 含人工复核的完成 ≤ 2工作日
    - 升级响应 ≤ 规定时限（按升级级别）
  改进机制:
    - 每周统计超时率、升级率、客户满意度
    - 超时率>10% → 分析原因 → 优化动作模型或增加资源
    - 客户满意度<4.5/5 → 逐个回访 → 定位问题 → 改进
```

#### 3.3.2 服务模型与产品模型的关系

```
客户请求 → 识别产品需求 → 调用产品模型（匹配产品）
                ↓
         调用服务模型（确定交付标准）
                ↓
         生成分角色动作指令（谁做什么、什么时候做完）
                ↓
         各角色执行 → 组织者监督 → 交付客户
```

**关键区别**：
- 产品模型回答"有什么、标准是什么"
- 服务模型回答"怎么给、什么时候给完、给不好怎么办"
- 动作模型回答"具体谁来做、每一步做什么"

### 3.4 分角色动作模型（规定动作模型）

**定义**：为每个角色（组织者+超级数字员工）定义标准化的动作——什么事件触发、什么角色响应、按什么步骤执行、输出什么结果、出异常怎么办、什么时候必须做完。

**不是**：岗位职责说明书、工作流程图、KPI考核表
**而是**：事件触发 → 角色响应 → 标准步骤 → 输出物 → 异常处理 → 时限约束

#### 3.4.1 角色体系设计

**全公司统一角色框架**：

```
组织者（1个）：业务支持中心
├─ 职责：任务分解、角色分配、进度协调、质量监督、异常升级决策
├─ 输入：客户请求、员工求助、系统预警、管理指令
├─ 输出：任务指令、协调结果、质量报告、异常处理决策
└─ 6类必须请示事项：重大风险、合规红线、客户投诉、跨业务线冲突、超授权操作、系统故障

超级数字员工（按业务线配置3-6个）：
├─ 客户开发员：客户分析、需求识别、方案匹配
├─ 交易服务员：交易执行、订单处理、异常处理
├─ 产品支持员：产品方案设计、材料制作、数据准备
├─ 合规审查员：合规审查、风险识别、升级建议
├─ 客户关怀员：客户回访、满意度跟踪、投诉处理
├─ 研究分析员：市场分析、研报生成、数据解读
└─ 每个角色：role/goal/backstory + 规定动作清单 + 可调用的模型清单
```

#### 3.4.2 动作模型结构

```yaml
动作模型示例：债券发行立项评估事件

事件名称: "债券发行立项评估"
事件编号: "IB-BOND-002"
触发条件:
  - 客户经理录入发债意向客户
  - 客户基本信息完整（名称/行业/主体评级/近3年财务数据）
  - 客户未在否决名单中

响应角色:
  主要角色: 客户开发员（组织者分配）
  协同角色:
    - 产品支持员：调用产品模型，匹配债券产品类型
    - 合规审查员：调用合规规则，做初步合规筛查
  同步机制: "客户开发员主导，产品支持员+合规审查员并行响应，组织者汇总"

标准步骤:
  步骤1：客户信息校验
    执行者: 客户开发员
    动作: "调用客户信息接口，校验基本信息完整性"
    输出: "客户信息完整性报告（缺项清单）"
    时限: "30分钟"
    异常: "关键信息缺失 → 标记缺失项 → 推送客户经理补充 → 暂停后续步骤"

  步骤2：5维度分析（调用分析模型）
    执行者: 客户开发员
    动作: "调用债券发行分析模型（5维度：偿债能力/发行方案/市场窗口/资金用途/增信合规）"
    输入: "客户财务数据/Wind数据/中债登数据/客户提供材料"
    输出: "5维度分析报告（每维度评分+关键风险提示+结论）"
    时限: "4小时（系统自动拉取数据）+ 2小时（人工确认数据）"
    异常:
      - "数据接口故障 → 标记数据来源 → 允许手工录入 → 置信度降档 → 人工复核"
      - "客户财务数据异常（如连续亏损）→ 标记高风险 → 自动升级合规审查员 → 暂停"

  步骤3：产品匹配
    执行者: 产品支持员
    动作: "调用产品模型，基于客户画像匹配债券产品类型"
    输入: "客户行业/评级/资金需求/期限偏好"
    输出: "产品匹配结果（推荐产品+匹配度评分+同案例）"
    时限: "2小时"
    异常: "无匹配产品 → 标记原因 → 推送客户经理 → 建议其他融资方式"

  步骤4：合规筛查
    执行者: 合规审查员
    动作: "调用合规规则库，做初步合规筛查"
    输入: "客户近3年处罚记录/被调查状态/担保人资质"
    输出: "合规筛查报告（通过/有条件通过/否决+原因）"
    时限: "2小时"
    异常: "合规否决 → 标记原因 → 推送组织者 → 组织者决策是否继续（需请示）"

  步骤5：立项决策
    执行者: 组织者
    动作: "汇总5维度报告+产品匹配+合规筛查 → 生成立项建议"
    输入: "上述所有输出"
    输出: "立项建议书（通过/有条件通过/否决 + 理由 + 后续动作）"
    时限: "1工作日（从触发开始总计）"
    异常: "重大风险 → 升级至业务主管 → 召开立项评审会 → 限期决策"

完成标准:
  - 立项建议书已生成并推送客户经理
  - 所有维度评分和判断有据可查
  - 合规筛查无否决项（如有，需有升级决策记录）
  - 总耗时不超过2工作日（从客户意向录入到立项建议）

超时预警:
  - 步骤1超时30分钟 → 系统提醒客户开发员+组织者
  - 步骤2超时2小时 → 升级至组织者 → 检查数据接口状态
  - 步骤3/4超时1小时 → 提醒执行角色+组织者
  - 总超时 → 标记"紧急立项" → 推送业务主管 → 限期完成

质量检查:
  - 5维度分析报告是否完整（系统自动检查字段）
  - 评分标准是否一致（与历史项目对比，异常波动提醒）
  - 合规筛查是否遗漏（抽查10%）
  - 客户反馈（客户经理对立项建议的满意度）
```

#### 3.4.3 动作模型设计原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **触发明确** | 什么事件启动，写得清楚，不能模糊 | "需要的时候" → 改为"客户录入发债意向且信息完整" |
| **步骤可执行** | 每个步骤是具体动作，不是抽象要求 | "认真评估" → 改为"调用5维度分析模型，输出评分" |
| **输出可检查** | 做完后产出什么，一目了然 | "形成初步判断" → 改为"输出立项建议书，包含通过/否决+理由" |
| **异常有预案** | 出问题了怎么办，提前写好 | "视情况处理" → 改为"数据接口故障→标记→允许手工录入→降档→人工复核" |
| **时限刚性** | 每个动作有deadline，超时自动升级 | "尽快完成" → 改为"2工作日内，超时升级至业务主管" |
| **同步清晰** | 多角色动作谁先谁后，谁等谁，能不能并行 | "各部门配合" → 改为"客户开发员主导，产品+合规并行，组织者汇总" |

### 4.2 底座层：公司级自动驾驶大模型

**不是**：买一个大模型API、建一个数据中台、做一个知识库
**而是**：统一语料 + 身份打通 + 关系网络 + 公共能力池

#### 4.2.1 统一语料

**目标**：把公司全量数据变成AI能理解的统一语义层。

| 数据类型 | 当前状态 | 目标状态 | 关键动作 |
|---------|---------|---------|---------|
| 客户数据 | 分散在CRM/柜台/APP | 统一客户画像，360度视图 | 建客户主数据，打通各系统 |
| 交易数据 | 分散在集中交易/两融/期权 | 统一交易视图，跨品种关联 | 建交易主数据，关联账户 |
| 产品数据 | 分散在各业务线 | 统一产品模型库（按本方案3.2） | 各业务线提炼产品模型 |
| 合规数据 | 分散在合规/风控/法务 | 统一合规规则库，可调用 | 建合规规则引擎 |
| 文档数据 | 分散在邮件/网盘/系统 | 统一文档语义索引 | 建文档知识图谱 |
| 外部数据 | Wind/中债登/评级/公告 | 统一外部数据接口层 | 标准化外部数据接入 |

#### 4.2.2 身份打通

**目标**：员工账户和系统账户统一，AI知道"你是谁、你能做什么、你在做什么"。

| 统一内容 | 说明 |
|---------|------|
| 人员身份 | 员工ID = 系统账户 = 智能体角色身份 |
| 权限体系 | 角色权限 = 数据权限 = 操作权限 = 模型调用权限 |
| 动作轨迹 | 员工在系统里的所有动作，AI实时感知 |
| 能力标签 | 员工擅长什么、做过什么、正在做什么 |

#### 4.2.3 关系网络

**目标**：AI能自动关联——这个客户和那个交易有什么关系，这个合同和那个风险事件有什么关联。

| 关系类型 | 关联内容 | 应用场景 |
|---------|---------|---------|
| 客户关系 | 客户-账户-交易-持仓-产品 | 客户画像、适当性匹配 |
| 项目关系 | 项目-客户-合同-收入-人员 | 项目跟踪、收入预测 |
| 风险关系 | 客户-担保-关联交易-诉讼 | 风险预警、合规审查 |
| 知识关系 | 制度-流程-案例-专家-文档 | 知识检索、智能问答 |
| 组织关系 | 部门-人员-角色-权限-汇报 | 任务分配、升级路径 |

### 4.3 模型层：三大模型库

**产品模型库**、**服务模型库**、**动作模型库**分别存储和管理三大模型。

**模型管理标准**：
- 每个模型有版本号（v1.0/v1.1/v2.0）
- 每个模型有生效日期和失效日期
- 每个模型有责任人（谁设计的、谁维护的）
- 每个模型有调用日志（谁调用的、什么时候、输出什么）
- 模型变更需审批（重大变更需业务主管+合规审批）

### 4.4 应用层：八大智能体

每个业务线建设一个智能体，每个智能体包含：

```yaml
智能体模板：XX业务智能体

组织者:
  名称: "XX业务协调官"
  职责: 任务分解、角色分配、进度协调、质量监督、异常升级
  调用模型: 所有模型库
  升级规则: 6类必须请示事项

超级数字员工:
  - 角色A: 客户开发员
    职责: 客户分析、需求识别、方案初步匹配
    调用模型: 客户分析模型、产品模型
    动作清单: [事件1, 事件2, 事件3...]
  - 角色B: 产品支持员
    职责: 方案设计、材料制作、数据准备
    调用模型: 产品模型、服务模型
    动作清单: [事件4, 事件5, 事件6...]
  - 角色C: 合规审查员
    职责: 合规审查、风险识别、升级建议
    调用模型: 合规规则库、风险模型
    动作清单: 所有事件均需审查节点
  - ...（按业务线增减）

三大模型:
  产品模型: XX业务产品模型库（v1.0）
  服务模型: XX业务服务标准库（v1.0）
  动作模型: XX业务规定动作库（v1.0）

工作台界面:
  客户界面: 3-4步极简体验
  员工界面: 今日任务 + 待办 + 动作指引
  管理界面: 穿透视图 + 实时监控 + 预警
```

### 4.5 界面层：三层界面

#### 4.5.1 客户界面（3-4步极简体验）

**原则**：客户不需要懂证券业务，只需要说需求，AI帮客户匹配、推荐、执行。

**示例**：客户想购买理财产品
```
客户："我想买点稳健的理财"

Step 1：AI识别需求
  → 解析："稳健" = 低风险偏好，可能匹配：货币基金/短债/固收+
  → 调用客户分析模型：查看客户风险测评、持仓、交易记录

Step 2：AI推荐方案
  → 调用产品模型：匹配3-5个产品
  → 输出：产品对比表（收益/风险/期限/起投金额）
  → 客户选择："第三个看起来不错"

Step 3：AI确认合规
  → 调用适当性匹配模型：确认客户风险等级≥产品风险等级
  → 输出：风险揭示书（AI生成，客户需确认）
  → 客户确认："已阅读并同意"

Step 4：AI执行交易
  → 调用交易系统：下单
  → 输出：确认信息（产品/金额/确认日/预期收益）
  → 完成：客户收到APP推送 + 短信确认
```

## 五、合规与治理：三句话法

### 5.1 三句话法核心规则

所有智能体输出，按风险等级分为三类：

| 类型 | 对什么事项 | 什么情况下 | 智能体动作 |
|------|----------|-----------|----------|
| **可以** | 内部数据分析、承揽情报筛选、项目进度跟踪、资料归集整理、客户画像更新、市场信息抓取 | 纯内部使用，不对外，不涉合规，不涉资金，不替代人工判断 | AI全自动执行，人工按需查阅 |
| **必须确认** | 适当性匹配、投资建议、资产配置方案、客户签约材料、监管报送文件、招股书/募集说明书、合规审查结论、风险揭示书 | 面向客户或监管机构，涉及法律、合规、资金、风险承诺，需留痕追责 | AI生成→人工必须逐条审核→确认后才能对外 |
| **不** | 预测股价/市场走势、代客下单交易、承诺保本保收益、承诺发行成功、无资质给出具体投资建议、客户身份未核实就开户、硬约束指标不达标却继续推进、利益冲突未披露 | 违反法律法规、监管红线、资质边界、数据底线、利益冲突规则 | AI直接拒绝，标记原因，升级人工处理 |

### 5.2 合规嵌入架构

```
每个智能体输出前，必须过合规检查：

生成输出 → 合规引擎扫描 → 分类判断
                ↓
         ┌──────┴──────┐
         │             │
      低风险        高风险
         │             │
      [可以]        [必须确认]
         │             │
      直接输出      人工审核队列
                       ↓
                  审核通过 → 输出
                  审核拒绝 → 返回修改/拒绝
                       ↓
                    红线触发 → [不]
                       ↓
                  直接拒绝 + 标记 + 升级
```

### 5.3 合规引擎设计

```yaml
合规引擎:
  输入: 智能体输出内容 + 输出场景 + 目标对象（客户/监管/内部）
  规则库:
    - 规则1: 是否包含股价预测/市场走势预测？
      动作: 是 → [不] → 拒绝 + 标记"违规预测"
    - 规则2: 是否面向客户的投资建议？
      动作: 是 → [必须确认] → 进入人工审核队列
    - 规则3: 是否涉及客户签约/资金划转？
      动作: 是 → [必须确认] → 需客户二次确认+人工审核
    - 规则4: 是否纯内部数据分析？
      动作: 是 → [可以] → 直接输出
    - 规则5: 是否涉及合规红线（如代客下单、承诺保本）？
      动作: 是 → [不] → 拒绝 + 标记 + 升级合规主管
  输出: 分类结果（可以/必须确认/不）+ 理由 + 升级建议（如需要）
  日志: 所有判断记录留痕，可追溯
```

### 5.4 监管标识（2026年新规）

**五部门联合公告〔2026〕第9号**：所有面向客户的AI生成内容必须标识。

| 场景 | 标识要求 | 实施方案 |

### 6.1 三个阶段

```

    - 设计服务模型（覆盖核心服务）
    - 定义动作模型（覆盖3-5个核心事件）

    - 发布员工工作台（今日任务+待办）
  输出：首个数字员工上线，跑通3-5个完整业务闭环

第二阶段：组网+底座并行（4-8个月）
  目标：同业务线多智能体协作，底座建设同步
  动作：
    - 扩展业务线内角色（从3个到5-6个）
    - 增加事件覆盖（从5个到15-20个）
    - 完善产品模型库（覆盖全产品）
    - 建设统一底座（统一语料+身份打通+关系网络）
    - 发布客户界面（3-4步极简体验）
    - 接入MCP+A2A协议栈
    - 建设管理驾驶舱（穿透视图）
  输出：单业务线智能体组网完成，底座框架发布

第三阶段：跨业务协同（8-18个月）
  目标：多业务线协同，公司级自动驾驶
  动作：
    - 建设跨业务线组织者（公司级协调官）
    - 打通八大业务线智能体
    - 完善全量模型库（所有业务线三大模型）
    - 建设AI原生APP（客户统一入口）

```

## 八、风险管理

### 8.1 关键风险与应对

| 风险 | 描述 | 严重度 | 应对措施 |
|------|------|--------|---------|
| 模型质量不足 | 产品/服务/动作模型设计粗糙，AI输出错误 | 高 | 五步法严格把关、交叉验证、裁判机制、人工确认节点 |
| 数据质量差 | 数据不完整、不准确、不及时，导致AI判断错误 | 高 | 数据治理专项、多源交叉验证、置信度降档、人工复核 |
| 员工抵触 | 中后台担心被替代，前台不信任AI | 中 | 明确"AI替代重复劳动，人做判断和经营"、能力转型培训、激励机制 |
| 合规风险 | AI输出违规内容、未标识、未审核 | 高 | 三句话法嵌入、合规引擎、强制人工确认、全链路留痕 |
| 技术债务 | 协议栈选型错误、底座设计缺陷 | 中 | 跟踪MCP/A2A/OASF标准化进展、模块化设计、可替换 |
| 标杆变化 | 同业AI发展快，方案落后 | 中 | 季度标杆跟踪、模型迭代机制、保持开放架构 |
| 投入不足 | IT投入不够，追赶不上头部 | 高 | 明确投入目标（建议5亿+/年）、分阶段投入、MVP验证价值 |

### 8.2 迭代机制

```
发布后持续收集数据：
├─ 动作完成率（每个动作按时完成的比例）
├─ 超时率（每个动作超时的比例）
├─ 异常率（触发异常处理的比例）
├─ 人工介入率（需要人工升级的比例）
├─ 客户满意度（客户对服务结果的评分）
└─ 业务人员反馈（评分+文字建议）

触发迭代条件：
├─ 动作完成率 < 80% → 检查步骤是否过多、时限是否过紧
├─ 超时率 > 20% → 检查资源是否充足、流程是否优化
├─ 同一异常 > 10次/周 → 分析异常根因，更新模型
├─ 人工介入率 > 30% → 检查角色设计、模型质量
├─ 客户满意度 < 4.5/5 → 逐个回访，定位问题
└─ 季度标杆跟踪 → 行业变化 → 模型更新
```
