> **版本**：v2.0  
> **日期**：2026-05-30

## 一、角色定义（先确定角色，再建模型）

## 三、业务标准模型（详细设计）

### 模型1：客户分析模型（底座级）

> **硬性标准**：客户分析模型不能少，是底座。基于身份证、账户、关系网络、社会分层四个维度构建。不要叫"高净值"，按社会阶层分类。

**目标角色**：客户经理（业务团队AI工作台自助调用）  
**核心诉求**：这个客户是谁？值不值得跟进？

**本质维度（4个，硬性标准）**：

| 维度 | 名称 | 讲人话的描述 | 客观数据 | 禁止 |
|------|------|-------------|---------|------|
| 1 | **身份证** | 客户是谁？实控人是谁？股权结构什么样？ | 工商信息、实控人、股权结构 | 模糊描述 |
| 2 | **账户** | 客户在华创的账户状态、资产状况、交易活跃度 | 账户数据、AUM、交易记录 | 叫"高净值" |
| 3 | **关系网络** | 我方哪位员工与客户哪位关键人有关系？发生过什么？ | 客户关系数据库、历史项目记录 | 主观评价 |
| 4 | **社会分层** | 客户在社会中处于什么位置？掌握什么资源？有什么影响力？ | 行业地位、影响力、资源层级 | AUM分层 |

**调用方式**（业务团队AI工作台）：
```
输入：客户名称（如"XX科技"）
输出：
  ├─ 身份证：实控人/股权结构/成立时间
  ├─ 账户：AUM/交易活跃度/产品持有情况
  ├─ 关系网络：我方触点/关键人/历史事件
  ├─ 社会分层：行业地位/资源层级/影响力
  └─ 跟进建议：值得/观望/暂缓 + 理由
```

**人工介入点**：关系网络维度需客户经理确认历史互动真实性。

---

### 模型2：业务分析模型

**目标角色**：客户经理（业务团队AI工作台自助调用）  
**核心诉求**：这个定增项目技术上能不能做？监管会不会卡？

**本质维度（4个）**：

| 维度 | 名称 | 讲人话的描述 | 客观数据 |
|------|------|-------------|---------|
| 1 | **发行规模大不大** | 打算融多少钱？相对于市值占比多少？市场能不能消化？ | 拟募集资金/总市值/同行业可比 |
| 2 | **价格定在哪** | 定价基准怎么选？折扣空间有多少？对现有股东稀释多少？ | 定价基准日/折扣率/股本稀释比例 |
| 3 | **时机对不对** | 现在发是不是好时候？监管环境松还是紧？市场窗口开没开？ | 近期监管政策/市场发行成功率/同行业案例 |
| 4 | **募投靠不靠谱** | 募投项目有没有明确用途？钱怎么花？能不能讲清楚故事？ | 募投项目说明书/可行性分析/行业前景 |

**调用方式**（业务团队AI工作台）：
```
输入：客户名称 + 拟发行规模 + 初步定价方案
输出：
  ├─ 发行规模大不大：规模合理性判断 + 可比案例
  ├─ 价格定在哪：定价建议区间 + 折扣影响分析
  ├─ 时机对不对：当前窗口判断 + 建议申报时间
  ├─ 募投靠不靠谱：项目可行性评估 + 故事完整性
  └─ 综合判断：可行/谨慎/建议暂缓 + 理由
```

**人工介入点**：时机判断涉及政策解读，需质控内核角色确认。

---

### 模型3：方案设计模型

**目标角色**：客户经理（业务团队AI工作台）+ 产品支持角色（业务支持AI工作台）  
**核心诉求**：给出一个可执行的定增方案框架。

**本质维度（3个）**：

| 维度 | 名称 | 讲人话的描述 | 输出标准 |
|------|------|-------------|---------|
| 1 | **怎么发** | 定价还是竞价？锁定期多长？面向谁发？ | 发行方式/锁定期/发行对象类型 |
| 2 | **多少钱** | 募资金额、定价区间、对股本的稀释比例 | 金额区间/价格区间/稀释比例 |
| 3 | **干什么** | 募投项目是什么？钱怎么分配？预期回报怎么讲？ | 项目清单/资金分配/回报测算 |

**调用方式**：
```
业务团队AI工作台（客户经理）：
  输入：客户分析结果 + 业务分析结果 + 客户诉求
  输出：方案框架（怎么发+多少钱+干什么）+ 风险提示

业务支持AI工作台（产品支持角色）：
  输入：客户经理提交的服务请求
  输出：完整方案建议书（含可比案例、定价测算、募投设计）
```

**人工介入点**：方案生成后必须经质控内核角色审核。

### 模型4：交叉验证模型（独立验证）

> **硬性标准**：交叉验证是独立模型类别，不是流程中的附加环节。一个智能体干活，一个智能体验证。

**目标角色**：质控内核角色（业务支持AI工作台执行）  
**核心诉求**：关键输出有没有问题？数据对不对？逻辑通不通？

**本质维度（3个）**：

| 维度 | 名称 | 讲人话的描述 | 验证标准 |
|------|------|-------------|---------|
| 1 | **合规验证** | 方案/材料是否符合监管要求？有没有踩红线？ | 法规比对/负面清单/历史处罚 |
| 2 | **数据验证** | 财务数据、定价数据、募投数据是否准确？有没有勾稽错误？ | 数据勾稽/来源校验/一致性检查 |
| 3 | **逻辑验证** | 方案逻辑是否自洽？不同模型输出是否一致？有没有悖论？ | 交叉比对/逻辑一致性/因果合理性 |

**调用方式**（业务支持AI工作台）：
```
触发条件：
  ├─ 方案设计完成后自动触发
  ├─ 材料制作完成后自动触发
  ├─ 发行定价决策前强制触发

输入：待验证的产出物（方案/材料/定价建议）
输出：
  ├─ 合规验证：✅/❌ + 具体问题清单
  ├─ 数据验证：✅/❌ + 数据疑点清单
  ├─ 逻辑验证：✅/❌ + 逻辑矛盾清单
  └─ 验证结论：通过/有条件通过/拒绝 + 修改建议

机制：
  - 产品支持角色（干活）→ 质控内核角色（验证）
  - 验证不通过 → 退回修改 → 再次验证
  - 验证通过 → 进入下一环节
```

**人工介入点**：验证结论必须由质控内核角色人工确认。

---

### 模型5：客户服务模型

**目标角色**：客户经理（业务团队AI工作台自助调用）  
**核心诉求**：项目做完了，客户关系怎么维护？下一个业务在哪？

**本质维度（3个）**：

| 维度 | 名称 | 讲人话的描述 | 客观数据 |
|------|------|-------------|---------|
| 1 | **项目有没有后遗症** | 发行后股价表现怎么样？投资者有没有亏钱？有没有投诉？ | 股价走势/投资者反馈/舆情监控 |
| 2 | **客户还缺什么** | 客户下一步还可能需要什么？并购？发债？股权激励？ | 客户业务规划/行业趋势/历史需求 |
| 3 | **关系还在不在** | 关键人有没有变动？有没有新的对接窗口？ | 人事变动/近期互动记录 |

**调用方式**（业务团队AI工作台）：
```
输入：客户名称 + 项目发行日期
输出：
  ├─ 项目后遗症：发行后表现 + 投资者反馈
  ├─ 客户还缺什么：潜在业务机会清单
  ├─ 关系还在不在：关键人状态 + 互动建议
  └─ 下一步动作：跟进计划建议
```

## 四、动作标准模型（按角色详细设计）

### 4.1 组织者角色动作

#### 动作1：任务分解与调度

| 要素 | 内容 |
|------|------|
| **动作名称** | 任务分解与调度 |
| **执行角色** | 组织者角色 |
| **输入条件** | 客户经理提交的服务请求 |
| **执行步骤** | 1. 意图识别（理解需求类型）<br>2. 任务分解（拆分为可执行动作）<br>3. 资源分配（匹配质控/产品支持角色）<br>4. 时限设定（明确每个动作时限）<br>5. 状态同步（实时通知客户经理） |
| **输出标准** | 任务清单（含执行人、时限、输入、输出） |
| **完成时限** | 实时（30分钟内完成分配） |
| **责任人** | 组织者角色（不能转包） |

#### 动作2：进度追踪与异常升级

| 要素 | 内容 |
|------|------|
| **动作名称** | 进度追踪与异常升级 |
| **执行角色** | 组织者角色 |
| **输入条件** | 各动作执行状态 |
| **执行步骤** | 1. 实时监控各动作进度<br>2. 超时预警（提前1日提醒）<br>3. 异常识别（卡住/争议/资源不足）<br>4. 升级处理（呼叫协同端/上报领导）<br>5. 结果反馈（完成后通知客户经理） |
| **输出标准** | 状态报告（实时同步业务团队AI工作台） |
| **完成时限** | 持续（全周期） |
| **责任人** | 组织者角色（不能转包） |

### 4.3 产品支持角色动作

#### 动作6：定增方案设计

| 要素 | 内容 |
|------|------|
| **动作名称** | 定增方案设计 |
| **执行角色** | 产品支持角色 |
| **输入条件** | 客户分析结果 + 业务分析结果 + 客户诉求 |
| **执行步骤** | 1. 调用方案设计模型 → 生成框架<br>2. 比对3-5个可比案例<br>3. 测算定价区间和稀释比例<br>4. 设计募投项目框架<br>5. 生成方案建议书<br>6. **提交质控内核角色交叉验证** |

| **责任人** | 产品支持角色（不能转包） |
| **人工介入点** | 方案确认需客户经理+质控内核共同确认 |

#### 动作7：募集说明书制作

| 要素 | 内容 |
|------|------|
| **动作名称** | 募集说明书制作 |
| **执行角色** | 产品支持角色 |
| **输入条件** | 方案确认 + 客户财务数据 + 募投项目资料 |
| **执行步骤** | 1. 抽取财务数据，填充财务章节<br>2. 生成募投项目描述<br>3. 填充法律合规章节<br>4. 生成风险因素章节<br>5. 格式校验、版本管理<br>6. **提交质控内核角色交叉验证** |

| **完成时限** | 5个工作日（初稿） |
| **责任人** | 产品支持角色（不能转包） |
| **人工介入点** | 质控内核审核、客户确认（必须人工） |

#### 动作8：发行执行

| 要素 | 内容 |
|------|------|
| **动作名称** | 发行执行 |
| **执行角色** | 产品支持角色 |

| **输出标准** | 发行结果报告（认购倍数、最终价格、配售名单） |
| **完成时限** | 按发行窗口执行 |
| **责任人** | 产品支持角色（不能转包） |
| **人工介入点** | 定价决策、配售方案必须人工确认 |

#### 动作9：持续督导

| 要素 | 内容 |
|------|------|

| **责任人** | 产品支持角色（不能转包） |

## 八、关键红线

### 8.1 客户分析底座红线
- **客户分析模型不能少**（硬性标准）
- **不要叫"高净值"**，按社会阶层分类
- 四个维度：身份证、账户、关系网络、社会分层

### 8.2 交叉验证红线
- 交叉验证是**独立模型类别**，不是附加环节
- 产品支持角色（干活）→ 质控内核角色（验证）
- 关键产出（方案/材料/定价）必须交叉验证通过

### 8.3 角色红线
- 业务支持中心 = 超级智能体
- 归纳四个以内主要角色
- 组织者角色是业务支持中心下面的角色之一
- 支持中心不管"人"，只管业务和动作

### 8.4 责任红线
- 动作责任人不能转包
- "委托尽调"被叫停
- 关键节点必须人工确认

### 8.5 输出红线
- 一页纸+一张图
- 不要长文本、不要PPT、不要主观画像
