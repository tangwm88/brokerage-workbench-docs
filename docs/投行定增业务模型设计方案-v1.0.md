## 一、角色定义：先定角色，再写模型

### 1.1 三端角色（定增业务场景）

| 端 | 角色名称 | 谁 | 在定增业务里干什么 | 核心诉求 |
|---|---|---|---|---|
| **员工端** | 投行业务人员（前台） | 承揽人、承做人员、保荐代表人 | 找项目、做方案、推发行、管客户 | 快速判断一个定增项目能不能做、怎么做、找谁做 |
| **业务支持端** | 投行支持中心 | 资本市场部、质量控制部、运营支持 | 方案设计、材料制作、合规审查、发行执行 | 按标准动作完成专业交付，不背锅 |
| **协同端** | 圆桌会议/跨部门协同 | 研究所、资管、财富、机构经纪、法律合规 | 提供研究、资金、客户、合规等专业支持 | 被需要的时候精准介入，不瞎掺和 |

## 二、业务标准模型（前端可调用的分析模型）

> **设计原则**：业务人员（员工端）直接调用的模型，3-5个本质维度，讲人话，输出可指导行动。

---

### 模型1：定增项目客户分析模型

**目标角色**：投行业务人员（承揽/承做）  
**核心诉求**：这个客户现在做定增，合不合适？切入点在哪？

**本质维度（3个）**：

| 维度 | 名称 | 讲人话的描述 | 客观数据来源 |
|---|---|---|---|
| 1 | **钱紧不紧** | 客户现在缺不缺钱？账上现金够不够？负债压力大不大？ | 货币资金/短债/经营现金流/应付账款账期 |
| 2 | **股性好不好** | 股价现在什么位置？市场认不认？流动性够不够？ | 股价/换手率/市值/近一年波动率 |
| 3 | **关系深不深** | 我们跟客户关键人（董事长/财务总监/董秘）有没有实质关系？ | 客户关系数据库、历史项目记录 |

**模型调用方式**：
```
输入：客户名称（如"XX科技"）
输出：
  ├─ 钱紧不紧：现金流紧张度评分 + 具体数据点
  ├─ 股性好不好：股价位置判断 + 市场流动性评估
  ├─ 关系深不深：我方触点梳理 + 关键人关系强度
  └─ 下一步建议：是否值得跟进 + 切入点建议
```

**人工介入点**：关系深不深维度涉及主观判断，需业务人员确认。

---

### 模型2：定增项目业务分析模型

**目标角色**：投行业务人员（承做）  
**核心诉求**：这个定增项目技术上能不能做？监管会不会卡？

**本质维度（4个）**：

| 维度 | 名称 | 讲人话的描述 | 客观数据来源 |
|---|---|---|---|
| 1 | **发行规模大不大** | 打算融多少钱？相对于市值占比多少？市场能不能消化？ | 拟募集资金/总市值/同行业可比 |
| 2 | **价格定在哪** | 定价基准怎么选？折扣空间有多少？对现有股东稀释多少？ | 定价基准日/折扣率/股本稀释比例 |
| 3 | **时机对不对** | 现在发是不是好时候？监管环境松还是紧？市场窗口开没开？ | 近期监管政策/市场发行成功率/同行业案例 |
| 4 | **募投靠不靠谱** | 募投项目有没有明确用途？钱怎么花？能不能讲清楚故事？ | 募投项目说明书/可行性分析/行业前景 |

**模型调用方式**：
```
输入：客户名称 + 拟发行规模 + 初步定价方案
输出：
  ├─ 发行规模大不大：规模合理性判断 + 可比案例
  ├─ 价格定在哪：定价建议区间 + 折扣影响分析
  ├─ 时机对不对：当前窗口判断 + 建议申报时间
  ├─ 募投靠不靠谱：项目可行性评估 + 故事完整性
  └─ 综合判断：可行/谨慎/建议暂缓 + 理由
```

**人工介入点**：时机判断涉及政策解读，需保代或合规人员确认。

---

### 模型3：定增方案生成模型

**目标角色**：投行业务人员（承做）+ 投行支持中心（方案设计员）  
**核心诉求**：给出一个可执行的定增方案框架。

**本质维度（3个）**：

| 维度 | 名称 | 讲人话的描述 | 输出标准 |
|---|---|---|---|
| 1 | **怎么发** | 定价还是竞价？锁定期多长？面向谁发？ | 发行方式/锁定期/发行对象类型 |
| 2 | **多少钱** | 募资金额、定价区间、对股本的稀释比例 | 金额区间/价格区间/稀释比例 |
| 3 | **干什么** | 募投项目是什么？钱怎么分配？预期回报怎么讲？ | 项目清单/资金分配/回报测算 |

**模型调用方式**：
```
输入：客户分析结果 + 业务分析结果 + 客户诉求
输出：
  ├─ 方案框架（怎么发+多少钱+干什么）
  ├─ 可比案例参考（3-5个同类项目）
  ├─ 风险提示清单
  └─ 下一步动作清单（需谁确认、需什么材料）
```

**人工介入点**：方案生成后必须经保代和合规审查员审核。

---

### 模型4：定增客户服务模型

**目标角色**：投行业务人员（承揽/承做）  
**核心诉求**：项目做完了，客户关系怎么维护？下一个业务在哪？

**本质维度（3个）**：

| 维度 | 名称 | 讲人话的描述 | 客观数据来源 |
|---|---|---|---|
| 1 | **项目有没有后遗症** | 发行后股价表现怎么样？投资者有没有亏钱？有没有投诉？ | 股价走势/投资者反馈/舆情监控 |
| 2 | **客户还缺什么** | 客户下一步还可能需要什么？并购？发债？股权激励？ | 客户业务规划/行业趋势/历史需求 |
| 3 | **关系还在不在** | 关键人有没有变动？有没有新的对接窗口？ | 人事变动/近期互动记录 |

**模型调用方式**：
```
输入：客户名称 + 项目发行日期
输出：
  ├─ 项目后遗症：发行后表现 + 投资者反馈
  ├─ 客户还缺什么：潜在业务机会清单
  ├─ 关系还在不在：关键人状态 + 互动建议
  └─ 下一步动作：跟进计划建议
```

---

### 模型5：定增项目准入模型（独立）

**目标角色**：投行支持中心（合规审查员）  
**核心诉求**：这个项目能不能接？有没有红线？

**本质维度（3个）**：

| 维度 | 名称 | 讲人话的描述 | 判断标准 |
|---|---|---|---|
| 1 | **客户有没有硬伤** | 有没有正在被处罚？财报有没有问题？实控人有没有风险？ | 负面清单/监管处罚/财务异常 |
| 2 | **项目有没有前科** | 这个客户之前做过定增吗？上次做得怎么样？ | 历史项目记录/发行后表现 |
| 3 | **我们能不能做** | 我们有没有利益冲突？团队有没有 capacity？ | 利益冲突排查/项目饱和度 |

**模型调用方式**：
```
输入：客户名称 + 拟发行规模
输出：
  ├─ 客户有没有硬伤：✅/❌ + 具体风险点
  ├─ 项目有没有前科：历史项目表现
  ├─ 我们能不能做：利益冲突结果 + 团队负荷
  └─ 准入结论：通过/有条件通过/拒绝 + 条件
```

**人工介入点**：准入结论必须由合规负责人人工确认。

## 三、动作标准模型（业务支持中心执行标准）

> **设计原则**：业务支持中心不管"人"，只管业务和动作。每个动作有明确标准、输入、输出、责任人。

---

### 动作1：定增项目立项材料准备

| 要素 | 内容 |
|---|---|
| **动作名称** | 定增项目立项材料准备 |
| **执行角色** | 投行支持中心 — 项目分析员 |
| **输入条件** | 客户名称、初步发行规模、业务人员提交的立项申请 |
| **执行步骤** | 1. 调用客户分析模型，生成客户画像<br>2. 调用准入模型，完成合规初筛<br>3. 调用业务分析模型，生成可行性评估<br>4. 整合生成立项报告初稿<br>5. 提交立项委员会 |
| **输出标准** | 完整立项材料包（含客户画像、可行性分析、风险提示、团队建议） |
| **完成时限** | 2个工作日 |
| **责任人** | 项目分析员（不能转包） |
| **人工介入点** | 立项委员会审议（必须人工） |

---

### 动作2：定增方案设计

| 要素 | 内容 |
|---|---|
| **动作名称** | 定增方案设计 |
| **执行角色** | 投行支持中心 — 方案设计员 |
| **输入条件** | 立项通过、客户诉求、初步发行规模、监管政策 |

| **输出标准** | 方案建议书（含发行方式、定价区间、募投项目、风险提示） |
| **完成时限** | 3个工作日 |
| **责任人** | 方案设计员（不能转包） |
| **人工介入点** | 保代审核、客户确认（必须人工） |

## 四、业务闭环：不同角色员工调用模型方案

### 4.1 三个闭环（平行可交叉，非线性串联）

```
闭环1：员工独立完成（简单判断）
  业务人员 → 调用客户分析模型 → 自己判断 → 决定跟不跟进
                          ↓
闭环2：员工→支持中心（标准服务）
  业务人员 → 提交服务请求 → 组织者分解 → 支持中心执行动作
                          ↓
闭环3：支持中心→圆桌会议（跨部门协同）
  支持中心 → 呼叫协同端 → 研究所/资管/机构经纪介入 → 返回专业支持
```

---

### 4.2 闭环1：员工独立完成（自助调用）

**适用场景**：项目前期线索筛选、客户初步判断、日常客户关系维护

**调用路径**：
```
业务人员（工作台/飞书）
  └─ 输入："分析一下XX科技的定增机会"
      └─ 调用客户分析模型 → 钱紧不紧/股性好不好/关系深不深
      └─ 输出：客户画像简报 + 跟进建议
  └─ 业务人员判断：值得跟进 → 进入闭环2
                不值得 → 记录放弃理由
```

**模型权限**：
| 模型 | 调用权限 | 输出限制 |
|---|---|---|
| 客户分析模型 | 所有业务人员 | 仅限内部参考，不可外传 |
| 业务分析模型 | 承做人员、保代 | 含监管判断，需留痕 |
| 客户服务模型 | 所有业务人员 | 客户关系信息，需脱敏 |

---

### 4.3 闭环2：员工→支持中心（服务化调用）

**适用场景**：立项申请、方案设计、材料制作、合规审查、发行执行

**调用路径**：
```
业务人员（提交服务请求）
  └─ 输入："准备XX科技定增立项材料，拟融资5亿"
      └─ 组织者（投行协调官）识别意图 → 分解动作
      └─ 匹配动作标准模型：立项材料准备
          ├─ 项目分析员执行：调用客户分析模型 + 准入模型
          ├─ 生成立项报告初稿
          └─ 提交立项委员会
      └─ 输出：立项材料包
  └─ 业务人员确认：提交立项委员会
```

**服务请求标准格式**：
```
请求类型：[立项/方案/材料/合规/发行/督导]
客户名称：XXX
发行规模：XX亿
紧急程度：[普通/加急]
特殊说明：[如有]
```

**动作匹配规则**：
| 请求类型 | 匹配动作 | 执行角色 | 标准时限 |
|---|---|---|---|
| 立项 | 定增项目立项材料准备 | 项目分析员 | 2个工作日 |
| 方案 | 定增方案设计 | 方案设计员 | 3个工作日 |
| 材料 | 募集说明书自动生成 | 材料制作员 | 5个工作日（初稿） |
| 合规 | 合规审查与内核把关 | 合规审查员 | 2个工作日 |
| 发行 | 簿记建档与发行执行 | 发行执行员 | 按窗口 |
| 督导 | 持续督导与投后服务 | 项目分析员 | 按监管周期 |

---

### 4.4 闭环3：支持中心→圆桌会议（跨部门协同）

**适用场景**：需要研究所、资管、机构经纪、法务等专业支持的复杂项目

**触发条件**：
1. 客户需要行业深度研究支持 → 呼叫研究所
2. 发行需要对接资金方 → 呼叫资管/机构经纪
3. 交易结构涉及法律创新 → 呼叫法务
4. 跨业务线资源协调 → 圆桌会议

**调用路径**：
```
支持中心（方案设计员）
  └─ 发现：需要研究所行业研究支持
      └─ 呼叫协同端（圆桌会议）
          └─ 研究所 → 输出行业研究报告
          └─ 返回支持中心
      └─ 方案设计员整合行业研究 → 完善方案
  └─ 输出：完整方案建议书
```

**协同请求标准格式**：
```
请求来源：投行支持中心-方案设计员
协同类型：[研究/资金/法律/客户]
项目：XX科技定增
需求描述：需要行业竞争格局分析，用于募投项目论证
期望输出：行业研究报告（含竞争格局、市场规模、趋势判断）
期望时限：3个工作日
```

## 六、关键设计原则与红线

### 6.1 维度红线（必须遵守）

- **硬性标准**：每个模型最多5个维度，最少3个
- **语言标准**：用"钱紧不紧"不用"现金流紧张度指标"，用"股性好不好"不用"市场流动性评估"
- **数据标准**：每个维度必须指向客观数据，不能靠主观感觉

### 6.2 责任红线（必须遵守）

- **不能转包**：项目责任人必须自己完成尽调，不能"委托尽调"给系统或他人
- **人定维度**：AI只做分析，维度是业务人员定义的，分析结果仅供参考
- **人工确认**：准入、合规审查、定价决策、配售方案必须人工确认

### 6.3 组织红线（必须遵守）

- **支持中心不管"人"**：支持中心只管业务和动作，人员组织属于HR和团队长
- **支持中心不背锅**：支持中心按标准动作执行，最终决策责任在前台业务人员
- **圆桌会议按需呼叫**：不主动介入，被呼叫时才响应

## 七、实施建议

### 7.1 第一阶段：模型验证（1-2周）

1. **选1-2个真实定增项目**，用上述模型跑一遍
2. **验证维度是否够用**：业务人员看一遍，能不能做出判断
3. **验证动作是否可执行**：支持中心试一遍，能不能按标准完成
4. **收集反馈**：哪里卡壳、哪里不够、哪里冗余

### 7.2 第二阶段：工具化（2-4周）

1. **把模型封装成Skill**：每个模型一个提示词，接入工作台
2. **把动作标准化**：每个动作一个流程卡片，支持中心按卡片执行
3. **跑通一个完整项目**：从客户分析到发行完成，全程用模型和动作

### 7.3 第三阶段：规模化（1-2个月）

1. **扩展到其他股权业务**：IPO、可转债、配股参照定增模型改造
2. **扩展到其他业务线**：债券、并购、财富参照同样方法论
3. **持续迭代**：每跑一个项目，优化一次模型

## 八、附录：模型提示词示例（工程级）

### 客户分析模型提示词（工程级）

```
【角色】定增项目客户分析助手
【目标】帮助投行业务人员判断一个客户是否适合做定增

【维度】（必须按这3个维度输出，不要增减）
1. 钱紧不紧：分析客户当前资金状况
   - 货币资金/短期借款比例
   - 经营活动现金流净额
   - 应付账款规模与账期
   - 银行贷款规模

2. 股性好不好：分析客户股票市场表现
   - 当前股价相对于近一年高低点位置
   - 日均换手率
   - 总市值规模
   - 近一年波动率

3. 关系深不深：分析我方与客户关系
   - 是否为华创证券客户关系
   - 我方哪位员工与对方哪位关键人有关系
   - 历史上发生了什么事

【输出格式】
- 每个维度给出：结论（一句话）+ 关键数据（3-5个）+ 判断（适合/谨慎/不适合）
- 最后给出：综合建议（跟进/观望/放弃）+ 下一步动作

【约束】
- 不讲废话，只给判断和数据
- 不使用"行业景气度""商业模式"等模糊术语
- 不给出投资建议，只做项目可行性判断
- 关系维度涉及隐私，仅内部可见
```
