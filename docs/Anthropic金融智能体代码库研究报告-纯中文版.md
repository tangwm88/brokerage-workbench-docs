## 一、仓库全景概览

### 1.1 定位与性质

Anthropic 官方开源的**金融服务智能体参考实现**，覆盖投资银行、股票研究、私募股权、财富管理、基金运营、合规运营六大垂直领域。

**核心特征**：
- **纯文本定义，无需编译** — 所有技能、智能体定义、工作流均以文本文件形式存在，无构建步骤
- **一份源码，两种部署** — 同一套技能与智能体定义，既可通过对话式插件安装，也可通过编程接口无头部署
- **11个数据连接器** — 对接标普智汇、FactSet、Daloopa、晨星、穆迪等主流金融数据提供商
- **Apache 2.0 开源协议**

## 二、核心设计范式分析

### 2.1 技能定义范式

每个技能是一个**自包含目录**，核心文件为技能定义文件：

```yaml
---
名称: 可比公司分析
描述: |
  构建机构级可比公司分析，包含经营指标、估值倍数和统计基准。
  
  **适用场景：**
  - 上市公司估值（并购、投资分析）
  - 与行业标杆对比绩效
  
  **不适用场景：**
  - 无可比上市公司的私营企业
  - 尚未产生收入的初创企业
---

# 可比公司分析

## ⚠️ 关键规则：数据源优先级（必读）

**必须遵循以下数据源优先级：**
1. **首选：专用数据接口** — 如标普智汇、FactSet、Daloopa 等数据接口可用，必须优先使用
2. **数据接口可用时，禁止使用网络搜索**
3. **仅当数据接口不可用时：** 使用彭博终端、证监会/交易所披露文件等权威来源
4. **绝不以网络搜索作为主要数据源** — 缺乏准确性、审计追溯和机构级可靠性

## 核心原则
**"先搭建正确结构，再让数据讲故事。"**

## 第一节：文档结构与设置
### 标题区块（第1-3行）
```
第1行：【分析标题】- 可比公司分析
第2行：【公司列表及代码】• 公司1（代码1）• 公司2（代码2）
第3行：截至【日期】| 金额单位为【百万/十亿】，每股数据和比率除外
```

## 第二节：经营统计与财务指标
### 核心列（以此为起点）
1. **公司** — 统一格式的名称
2. **收入** — 规模指标（最近12个月、季度或年度，视场景而定）
3. **收入增速** — 同比增速
4. **毛利润** — 收入减营业成本
5. **毛利率** — 毛利润/收入（基础盈利能力）
6. **息税折旧前利润** — 扣除利息、税金、折旧、摊销前利润
7. **息税折旧前利润率** — 息税折旧前利润/收入（运营效率）

### 可选补充（根据行业/目的选择）
- **季度与最近12个月对比** — 如季节性影响显著
- **自由现金流** — 资本密集型或软件服务企业
- **自由现金流利润率** — 自由现金流/收入（现金创造效率）
- **净利润** — 成熟盈利企业
- **营业利润** — 折旧摊销差异大的企业
- **资本支出指标** — 重资产行业
- **四十法则** — 软件企业专用（增速% + 利润率%）
- **自由现金流转化率** — 盈利质量分析（进阶）

## ⚠️ 关键规则：公式替代硬编码 + 分步验证

**公式优先，禁止硬编码：**
- 每个衍生数值（利润率、倍数、统计量）必须是引用输入单元格的公式 — 绝不可粘贴预计算数值
- 使用 Python/openpyxl 生成表格时：写入 `=E7/C7`（公式字符串），不可写入 `0.687`（计算结果）
- 唯一可硬编码的值：原始输入数据（收入、股价等）— 且每个硬编码值必须有批注标注来源
- 原因：模型必须在输入变化时自动更新。硬编码利润率是潜藏的错误。

**与用户分步验证：**
- 搭建结构后 → 先展示标题布局，确认后再填充数据
- 输入原始数据后 → 展示输入区块，确认来源和期间后再写公式
- 构建经营指标公式后 → 展示计算出的利润率，与用户合理性校验后再进入估值
- 构建估值倍数后 → 展示倍数并确认合理性后再添加统计量
- **禁止从头到尾一次性构建后再展示** — 尽早发现错误

## 第三节：估值倍数与投资指标
### 核心估值列（以此为起点）
1. **公司** — 与经营部分顺序一致
2. **市值** — 当前市场估值
3. **企业价值** — 市值 ± 净负债/现金
4. **企业价值/收入** — 市场为每1元收入支付的价格
5. **企业价值/息税折旧前利润** — 市场为每1元利润支付的价格
6. **市盈率** — 股价相对净利润

### 交叉引用规则
**关键：** 估值倍数必须引用经营指标部分。同一份原始数据不可输入两次。如收入在C7，则企业价值/收入公式必须引用C7。

## 输出检查清单
交付可比分析前，验证：
- [ ] 所有公司真正可比
- [ ] 数据期间一致
- [ ] 公式引用单元格，非硬编码值
- [ ] 所有硬编码输入单元格有批注（来源或假设说明）
- [ ] 统计量包含至少5个指标（最大、75分位、中位、25分位、最小）
- [ ] 合理性检查通过（利润率逻辑、倍数合理）
- [ ] 日期戳为当前（"截至【日期】"）
```

**关键设计特征：**

| 特征 | 说明 | 与我方对比 |
|------|------|-----------|
| **元信息头部** | 名称 + 描述（含适用/不适用场景） | 类似，我方技能定义也采用此格式 |
| **分层结构** | 概述 → 核心原则 → 分步执行 → 检查清单 | 我方五步法更宏观，此范式更微观（单任务级） |
| **关键规则标记** | 用 ⚠️ 标记不可违背的规则（数据源优先级、公式规范） | **可借鉴**：将合规/质量红线显性化标注 |
| **反模式清单** | 明确列出"绝不可做"的行为 | **可借鉴**：将常见错误模式作为反面教材写入技能 |
| **检查清单** | 交付前必须逐项勾选的清单 | **可借鉴**：嵌入动作模型的验收环节 |
| **引用文件** | 引用子目录存放支撑材料（公式、模板、术语） | **可借鉴**：将参考资料与主技能分离，便于更新 |

### 2.2 智能体定义范式

**智能体定义 = 系统提示词 + 技能清单 + 工具配置 + 子代理列表**

以投行推介代理为例：

```markdown
# 投行推介代理

## 交付成果
给定目标公司名称和战略情况，交付：
1. **Excel估值工作簿** — 可比公司、先例交易、现金流折现、足球场汇总图。每个输出单元格均为可追溯至输入的实时公式。
2. **推介演示文稿** — 基于银行模板：情况概述、公司快照、估值汇总（足球场）、可比详情、先例详情、示意流程。每个图表绑定至工作簿的命名范围。

## 工作流程
1. **明确需求。** 确认目标、行业、情况。识别5-8个最相关的可比公司和5-10个先例交易。
2. **撰写情况概述。** 调用行业概览技能，起草公司快照和战略逻辑叙述。
3. **提取数据。** 使用数据接口获取交易倍数、先例交易数据和最新披露文件。
4. **展开可比公司。** 调用可比分析技能，以一致的指标定义和异常值标记列出可比公司和先例交易。
5. **搭建并购案例。** 调用杠杆收购模型技能，构建示意性杠杆收购。
6. **构建模型其余部分。** 调用现金流折现模型和三表模型；遵循审计规范。
7. **生成足球场图。** 每种方法的最小/中位/最大值 — 可比、先例、现金流折现、杠杆收购 — 含当前价格标记。
8. **填充演示文稿。** 调用推介演示文稿技能，基于银行模板填充。
9. **演示文稿质检。** 调用投行演示文稿质检 — 验证合计数一致、脚注完整、日期统一。

## 防护栏
- **禁止对外通讯。** 本智能体无邮件或消息工具。
- **每个数字必须标注来源。** 如倍数或先例无法从数据接口或披露文件获取，标记为"【未标注来源】"而非估算。
- **Excel模型构建后和演示文稿生成后必须暂停并提交审核。** 银行家批准每件交付物后方可继续。

## 使用的技能
行业概览 · 可比分析 · 杠杆收购模型 · 现金流折现模型 · 三表模型 · Excel审计 · 推介演示文稿 · 投行演示文稿质检 · 演示文稿刷新
```

**关键设计特征：**

| 特征 | 说明 | 与我方对比 |
|------|------|-----------|
| **触发条件** | 由用户显式调用 | 我方规定动作也有触发条件（事件-触发-步骤） |
| **步骤链** | 9步线性流程，每步调用一个技能 | 类似我方动作模型的"步骤"，但更细粒度 |
| **技能调用** | 用反引号标注技能名称，明确依赖关系 | 我方动作模型也有"调用模型"环节 |
| **质量关卡** | 第7步后和第9步后必须人工审核 | **可借鉴**：在关键节点设置"停止并等待确认" |
| **防护栏** | 明确列出禁止事项（禁止对外通讯、必须标注来源） | 类似我方三句话法（可以/必须确认/不可以） |

## 三、与我方方法论的对照分析

### 3.1 架构层对照

| 维度 | Anthropic 金融智能体 | 我方现状 | 差距/机会 |
|------|---------------------|---------|----------|
| **技能定义** | 技能定义文件（元信息头部 + 指令） | 技能定义文件（类似格式） | **基本持平** |
| **技能分层** | 垂直技能（源）→ 智能体插件（打包） | 技能目录（独立） | **需改进**：缺少"源→打包"同步机制 |
| **智能体定义** | 代理系统提示词 + 工作流 | 规定动作模型（事件-触发-步骤-输出-异常-时限） | **互补**：Anthropic 偏"单任务流"，我方偏"规定动作框架" |
| **子代理编排** | 可调用代理（配置声明 + 交接事件） | 未明确设计 | **差距**：我方需设计子代理通信协议 |
| **数据连接** | 模型上下文协议标准（声明式配置） | 未标准化（各业务线自行对接） | **差距**：建议评估协议兼容性 |
| **部署模式** | 对话式插件 + 编程接口（双模式） | 工作台化（前端调用） | **差距**：缺少无头部署能力 |
| **安全分层** | 只读层/编排层/写入层隔离 | 三句话法（合规边界） | **互补**：Anthropic 偏"技术隔离"，我方偏"合规规则" |
| **质量校验** | 清单校验 + 格式校验 + 预提交钩子 | 交叉验证报告（人工评审） | **差距**：缺少自动化校验工具链 |

### 3.2 业务覆盖对照

| 业务领域 | Anthropic 覆盖 | 我方覆盖 | 可借鉴内容 |

### 3.3 方法论细节对照

#### A. 五步法 vs Anthropic 工作流

| 五步法（我方） | Anthropic 对应 | 评价 |
|--------------|---------------|------|
| 第一步：业务目标（经营什么？关键动作？衡量指标？） | 隐含在智能体的"交付成果"和"明确需求" | Anthropic 缺少显式的业务目标蒸馏环节 |

| 第四步：模型定义（角色体系 + 分析模型 + 动作模型） | 代理系统提示词 + 技能 | **基本对齐**：我方更强调"角色-模型"关系，Anthropic 更强调"技能-工具"配置 |
| 第五步：交叉验证（自洽、标杆、可执行、裁判挑刺） | 清单校验 + 格式校验 + 人工审核关卡 | **互补**：Anthropic 有自动化校验工具，我方有人工裁判机制 |

#### B. 三句话法 vs Anthropic 防护栏

| 我方三句话法 | Anthropic 防护栏 | 融合建议 |
|------------|-----------------|---------|
| **可以**：内部数据分析、客户分析更新 → 智能体全自动 | "禁止对外通讯" | **对齐**：双方都限制了智能体的对外行为边界 |
| **必须确认**：投资建议、合规审查 → 智能体生成→人工审核→确认 | "关键节点暂停并提交审核" | **对齐**：双方都设置了人工确认关卡 |
| **不可以**：预测股价、代客下单 → 智能体直接拒绝 | "每个数字必须标注来源" | **互补**：我方是"拒绝型"边界，Anthropic 是"标记型"边界 |

#### C. 禁用词语表 vs Anthropic 语言风格

Anthropic 的技能文件中：
- 使用具体动词（"构建"、"填充"、"核对"）
- 避免互联网黑话（无"对齐"、"场景"、"逻辑"）
- 具体化描述（"收入（最近12个月）"而非"收入指标"）
- 白话表达（"市场为每1元收入支付的价格"解释企业价值/收入）

**结论**：Anthropic 的语言风格与我方禁用词语表高度一致，可作为参考标杆。

#### 3. 反模式清单

**来源**：投资银行/推介演示文稿/技能定义

```markdown
### 反模式一：将数据填入占位框
**现象：** 模板中有彩色指示框（黄色、橙色等）含指导文字。模型替换文字但保留彩色框。
**错误原因：** 彩色框本身就是占位符。
**识别方法：** 如填充后的幻灯片有大面积彩色矩形框内填充数据文字，即犯了此错误。

### 反模式二：文本"表格"
**现象：** 模型用分隔符（|、制表符、空格）创建类似表格的内容，而非真正的表格对象。
**错误原因：** 这不是表格。列永远无法正确对齐，无法一致格式化。
**识别方法：** 如你在输入 | 字符或依赖空格/制表符创建列，你是在创建文本而非表格。
```

**借鉴价值**：
- 将**常见错误模式**作为反面教材写入技能
- 每个反模式配"识别方法"（如何识别）+ "正确做法"
- **可直接用于我方**：债券募集书常见错误、适当性匹配常见误操作、客户分析常见偏差

## 七、结论

Anthropic 的金融智能体仓库是一份**高质量的金融智能体参考实现**，其核心价值在于：

1. **技能定义的工程化**：元信息头部 + 指令文档 + 引用文件 + 检查清单，形成完整的"技能即代码"范式
2. **安全设计的分层化**：只读层 → 编排层 → 写入层的三层隔离，尤其适用于处理不可信外部文档
3. **数据连接的标准化**：模型上下文协议统一对接 11 个金融数据提供商，声明式配置替代代码集成
4. **质量控制的自动化**：清单校验 + 格式校验 + 预提交钩子，形成从开发到部署的完整工具链

**对我方的最大启示**：
- 我们的**五步法 + 蒸馏智能体 + 规定动作**方法论在"业务理解深度"上优于 Anthropic（他们有流程但缺少业务目标蒸馏环节）
- 但在**工程化落地**上，Anthropic 领先一步（协议标准、部署脚本、校验工具链、安全分层）
- **建议策略**：方法论保持我方优势（五步法、三句话法、禁用词表），工程实现借鉴 Anthropic（协议兼容、子代理编排、自动化校验）
