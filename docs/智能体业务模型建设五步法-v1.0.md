## 二、Step 1：业务目标

### 2.1 什么是业务目标

业务目标回答三个问题：
1. **经营什么** —— 这条业务线的核心产出是什么？
2. **关键动作** —— 为了产出，必须做哪些事？
3. **衡量什么** —— 怎么判断做得好还是不好？

### 2.2 操作方法

```
Step 1.1: 识别核心业务
  └─ 这条业务线最终交付什么？（不是过程，是结果）
  └─ 例：投行债券发行 → 交付"成功发行的债券项目"
  └─ 例：零售客户服务 → 交付"持续活跃且满意的高价值客户"

Step 1.2: 拆解关键动作
  └─ 从交付倒推：为了交付这个结果，必须做什么？
  └─ 每个关键动作再拆：触发条件 → 执行步骤 → 输出物
  └─ 例：债券发行关键动作：线索获取 → 立项评估 → 方案设计 → 材料制作 → 监管申报 → 发行承销 → 后督管理

Step 1.3: 定义目标边界
  └─ 什么属于这条业务线？什么不属于？
  └─ 和其他业务线的交界在哪？会不会冲突？
  └─ 例：债券发行 vs 定增 —— 都是投行项目，但客户对接人可能相同，需要明确分工

Step 1.4: 设定衡量指标
  └─ 每个关键动作至少1个可衡量的指标
  └─ 指标必须可获取、可验证，不能是"感觉"
  └─ 例：立项评估 → "立项通过率"、"从线索到立项的平均天数"
```

### 2.3 输出模板

```yaml
业务线名称: "投行债券发行业务"
核心业务目标:
  目标1:
    名称: "成功发行债券"
    定义: "完成从项目线索到债券成功发行的全周期交付"
    关键动作:
      - 线索获取与核实
      - 立项评估（5维度分析）
      - 发行方案设计
      - 申报材料制作
      - 监管申报与反馈回复
      - 发行定价与簿记建档
      - 发行后督导
    衡量指标:
      - "年度发行项目数量"
      - "发行成功率（申报通过/总申报）"
      - "平均发行周期（立项到发行）"
    边界: "不包含银行间市场非金融企业债务融资工具（短融/中票/PPN）"
    
  目标2:
    名称: "项目质量控制"
    定义: "确保每个发行项目材料真实、风险可控、合规达标"
    关键动作:
      - 尽职调查质量把关
      - 募集说明书数据核查
      - 合规风险审查
      - 质控内核评级
    衡量指标:
      - "质控评级A级项目占比"
      - "监管反馈问题数量（均值）"
      - "发行后合规事件数量"
    边界: "不包含项目执行中的日常客户关系维护（归属客户服务）"

目标间关系:
  依赖:
    - "目标2质量通过 → 目标1才能申报"
  协同:
    - "目标1线索获取 → 发现客户需求 → 回流客户服务"
    - "目标2质控发现 → 反馈客户经理 → 改进尽调"
```

### 2.4 常见错误

| 错误 | 正确做法 |
|------|---------|
| "提升服务质量"（模糊） | "客户满意度评分≥4.5/5"（可衡量） |
| "加强客户覆盖"（过程导向） | "核心客户季度拜访覆盖率100%"（结果导向） |
| "做好风险管理"（无边界） | "发行项目合规事件0起"（有边界） |
| 关键动作列10个 | 压缩到5-7个，合并同类项 |

## 三、Step 2：标杆分析

### 3.1 什么是标杆分析

标杆分析不是"看看别人怎么做"，而是：
1. **找标杆** —— 谁在这一点上做得最好？为什么？
2. **拆做法** —— 他具体怎么做的？每一步是什么？
3. **评差距** —— 我们差在哪？能不能学？需不需要改？
4. **定借鉴** —— 哪些可以直接用？哪些需要调整？哪些不能学？

### 3.2 操作方法

```
Step 2.1: 识别标杆对象
  └─ 同业标杆：直接竞争对手（华泰/国泰海通/中信建投）
  └─ 跨业标杆：不同行业但有相似逻辑（华住集团/摩根士丹利/Betterment）
  └─ 国际标杆：海外市场先行者（高盛/摩根大通/BlackRock）
  └─ 选择标准：信息可获取、做法可验证、逻辑可迁移

Step 2.2: 收集标杆信息
  └─ 公开渠道：年报/公告/发布会/新闻报道/学术论文
  └─ 行业报告：Gartner/IDC/麦肯锡/甲子光年
  └─ 招聘网站：JD分析（招什么人、要什么技能）
  └─ 产品体验：亲自用对方产品（APP、工作台）
  └─ 政策文件：监管披露、合规要求

Step 2.3: 拆解标杆做法
  └─ 他的角色怎么设的？和我们的组织者+超级数字员工有什么异同？
  └─ 他的分析模型有几个维度？每个维度回答什么问题？
  └─ 他的动作是怎么标准化的？有没有触发-步骤-输出-异常-时限？
  └─ 他的数据从哪来？系统对接还是手工录入？
  └─ 他的合规怎么做？人工确认节点在哪？

Step 2.4: 差距分析与借鉴决策
  └─ 差距矩阵：我们在哪落后？差多远？
  └─ 借鉴可行性：技术/资源/合规/文化，哪个是瓶颈？
  └─ 借鉴决策：[直接采用]/[调整后采用]/[暂不采用]/[反面教材]
```

### 3.3 输出模板：标杆分析表

```markdown
| 维度 | 标杆对象 | 标杆做法 | 我方现状 | 差距 | 借鉴决策 | 备注 |
|------|---------|---------|---------|------|---------|------|
| 角色体系 | 华泰证券 | AI选股/盯盘/交易助手+客户经理 | 组织者+超级数字员工框架已设计，未落地 | 概念领先，落地滞后 | [调整后采用] 保持组织者+数字员工框架，学习华泰的AI助手命名和具象化 | 华泰已上线AI涨乐APP |
| 分析模型 | 中信建投 | 智能条件选股/全维度市场搜索/财务数据解读/实时行业动态/全面深度分析 | 5维度模型已有，具体指标已设计 | 方法论对齐，细节待验证 | [直接采用] 5维度框架与中信建投五大功能思路一致，继续深化 | 中信建投蜻蜓翼答5月上线 |
| 动作标准化 | 华住集团 | 入住SOP：预订→确认→入住→服务→离店，每个动作有明确标准 | 规定动作模型v1.0已设计，未大规模验证 | 框架对齐，行业场景待适配 | [调整后采用] 华住的酒店SOP逻辑迁移到证券业务，调整触发条件和异常处理 | 华住是服务标准化标杆 |
| 数据对接 | 国泰海通 | 君弘灵犀APP内部打通全量数据，AI客服替代人工60%+ | 数据分散在Wind/中债登/内部系统，未统一 | 数据整合差距大 | [暂不采用] 先完成单业务线数据打通，再考虑全量 | 国泰海通投入大、周期长 |
| 合规机制 | 摩根士丹利 | 所有AI输出需顾问审核，Human-in-the-loop | 关键节点人工确认已设计 | 思路一致，执行细节待明确 | [直接采用] 保持人工确认节点设计，细化确认触发条件 | 符合国内监管要求 |
| 技术栈 | Google/Anthropic | MCP+A2A+ADK三层协议，150家机构生产采用 | 规划中，未接入 | 落后1-2个季度 | [调整后采用] 2026年Q3前完成MCP+A2A预埋，ADK编排层视情况引入 | 技术窗口期在收窄 |
```

### 3.4 标杆分析的四个原则

| 原则 | 说明 |
|------|------|
| **可验证** | 标杆做法必须有公开信息来源，不能是"听说" |
| **可迁移** | 标杆的行业/场景差异太大，不能直接搬 |
| **不盲从** | 标杆也有问题（如华泰AI荐股合规风险），要学优点、避坑 |
| **持续跟踪** | 标杆不是静态的，每季度更新一次标杆库 |

## 四、Step 3：数据关联

### 4.1 什么是数据关联

数据关联回答三个问题：
1. **业务动作需要什么数据？** —— 每个步骤要什么输入？
2. **数据在哪？** —— 存在哪个系统？什么格式？
3. **怎么拿到？** —— API对接？手工录入？文件上传？公开抓取？

**核心原则**：如果数据拿不到，这个动作就是纸上谈兵。

### 4.2 操作方法

```
Step 3.1: 拆解每个动作的数据需求
  └─ 列出每个动作的执行步骤
  └─ 每个步骤需要什么数据？（字段名、数据类型）
  └─ 数据是必需还是可选？

Step 3.2: 盘点数据源
  └─ 内部系统：CRM/ERP/Wind/中债登/评级机构/内部数据库
  └─ 外部公开：交易所公告/工商信息/裁判文书/评级报告
  └─ 手工输入：客户经理录入/客户提供/会议纪要
  └─ 生成数据：系统计算/模型推断/规则判断

Step 3.3: 评估数据获取路径
  └─ 已有API：直接对接，优先级最高
  └─ 无API有界面：考虑RPA或手工录入
  └─ 无系统：手工录入，但要限次数（不能每个动作都靠手工）
  └─ 外部公开：爬虫/订阅服务/定期更新

Step 3.4: 建立数据关联矩阵
  └─ 动作 → 数据需求 → 数据源 → 获取路径 → 自动/手工比例
  └─ 目标是：80%自动获取，20%手工录入
```

### 4.4 数据关联的三个检查点

| 检查点 | 检查内容 | 不通过怎么办 |
|--------|---------|-------------|
| **数据可得性** | 这个数据现在能不能拿到？ | 拿不到 → 改动作设计，或先建手工流程 |
| **数据准确性** | 数据源可靠吗？会不会出错？ | 不可靠 → 多源交叉验证，或标记置信度 |
| **数据时效性** | 数据更新频率够吗？ | 不够 → 评估影响，标记时效风险 |

## 五、Step 4：模型定义

### 5.1 什么是模型定义

模型定义是把业务目标、标杆经验、数据基础，转化为**可执行的结构化模型**。包含三个层面：

1. **角色体系** —— 谁来做？（组织者+超级数字员工）
2. **业务分析模型** —— 怎么判断？（3-5维度，每个维度回答一个核心问题）
3. **规定动作模型** —— 怎么执行？（事件-触发-步骤-输出-异常-时限）

### 5.2 操作方法

```
Step 4.1: 定义角色体系
  └─ 组织者：1个，负责统筹、协调、决策升级
  └─ 超级数字员工：3-6个，覆盖业务全链条
  └─ 每个角色：role/goal/backstory + 规定动作清单
  └─ 检查：角色间有无重叠？有无遗漏？协作关系是否清晰？

Step 4.2: 设计业务分析模型
  └─ 先定模型的目标角色（只能有一个主要调用者）
  └─ 再提炼核心诉求（一句话说清这个模型解决什么问题）
  └─ 围绕诉求定义3-5个本质维度
  └─ 每个维度：核心问题 → 分析框架 → 判断标准 → 输出格式
  └─ 检查：维度间有无逻辑悖论？输出是否有动作指向？

Step 4.3: 设计规定动作模型
  └─ 识别业务事件（什么情况下启动一套动作？）
  └─ 定义响应角色（哪些角色需要同步响应？）
  └─ 设计标准动作（触发 → 步骤 → 输出 → 异常 → 时限）
  └─ 定义同步机制（谁先谁后？谁等谁？能不能并行？）
  └─ 设定完成标准（怎么算做完了？）
  └─ 设计超时预警（多久没做完要提醒谁？）

Step 4.4: 模型间衔接设计
  └─ 分析模型和动作模型怎么衔接？
  └─ 不同角色的模型怎么衔接？
  └─ 不同业务线的模型怎么衔接？
```

### 5.3 模型定义检查清单

#### 角色体系检查

| 检查项 | 合格标准 | 常见问题 |
|--------|---------|---------|
| 角色数量 | 组织者1 + 超级数字员工3-6 | 太多（>7）或太少（<3） |
| 角色边界 | 每个角色职责不重叠 | "这个活归谁？"说不清楚 |
| 升级路径 | 每个角色知道升级给谁 | 异常时不知道找谁 |
| 请示边界 | 组织者明确6类需请示事项 | 大事小事都请示，或大事没人管 |

#### 业务分析模型检查

| 检查项 | 合格标准 | 常见问题 |
|--------|---------|---------|
| 维度数量 | 3-5个，不能再多 | 贪多，列了8-10个 |
| 角色唯一 | 每个模型有明确的目标角色 | "这个模型谁用？"不清楚 |
| 讲人话 | 维度描述不用术语 | "现金流覆盖率"写成"偿债能力覆盖倍数" |
| 有动作 | 每个维度分析后必须有下一步 | 分析完没有动作指向 |
| 无悖论 | 维度间不自相矛盾 | 维度1说客户价值高，维度4说客户随时会流失 |

#### 规定动作模型检查

| 检查项 | 合格标准 | 常见问题 |
|--------|---------|---------|
| 触发明确 | 什么事件启动，写得清楚 | "需要的时候""视情况而定" |
| 步骤可执行 | 每个步骤是具体动作 | "认真评估""做好沟通" |
| 输出可检查 | 做完后产出什么，一目了然 | "形成初步判断" |
| 异常有预案 | 出问题了怎么办，提前写好 | "视情况处理" |
| 时限刚性 | 每个动作有deadline | "尽快完成""抓紧推进" |
| 同步清晰 | 多角色动作谁先谁后 | 并行变串行，互相等 |

## 六、Step 5：交叉验证（裁判验证）

### 6.1 什么是交叉验证

交叉验证不是"自己检查自己"，而是：
1. **内部自洽验证** —— 模型内部逻辑是否通顺？
2. **标杆对比验证** —— 和行业标杆比，差在哪？合理吗？
3. **可执行性验证** —— 拿一个真实案例跑一遍，能跑通吗？
4. **独立裁判验证** —— 找一个不参与设计的人/团队，用批判视角挑刺

### 6.2 操作方法

```
Step 5.1: 内部自洽验证
  └─ 检查角色间是否有职责重叠或遗漏
  └─ 检查分析模型维度间是否有逻辑矛盾
  └─ 检查动作模型步骤是否闭环（从触发到完成）
  └─ 检查模型间衔接是否顺畅

Step 5.2: 标杆对比验证
  └─ 角色设计：和标杆的角色体系对比，差异是否合理？
  └─ 模型维度：和标杆的分析框架对比，是否遗漏关键维度？
  └─ 动作标准：和标杆的SOP对比，我们的要求更松还是更紧？
  └─ 合规策略：和标杆的合规机制对比，是否有漏洞？

Step 5.3: 可执行性验证
  └─ 选择1-2个真实历史项目
  └─ 用新设计的模型"模拟复盘"
  └─ 记录：哪些步骤跑不通？哪些数据拿不到？哪些判断做不出？
  └─ 评估：整体跑通率是多少？

Step 5.4: 独立裁判验证
  └─ 找不参与设计的同事（最好是业务专家或一线人员）
  └─ 给他们模型，让他们用
  └─ 收集反馈：哪里看不懂？哪里觉得不对？哪里执行不了？
  └─ 裁判没有义务说好话，越刻薄越好
```

```markdown

### 二、标杆对比验证

### 三、可执行性验证（模拟复盘：XX公司2025年公司债发行项目）

| 事件 | 模拟步骤 | 结果 | 问题 |
|------|---------|------|------|
| 事件1：线索录入 | 客户经理录入XX公司债券发行意向 | ✅ 完成 | 无 |
| 事件2：立项评估 | 调用5维度模型，数据从Wind/中债登拉取 | ⚠️ 部分完成 | "已发行债券余额"需中债登接口，目前未对接 → 改为手工录入（影响效率但不影响判断） |

| 事件4：材料准备 | 客户经理收集材料，产品支持制作文件 | ⚠️ 部分完成 | 募投项目备案号需客户提供，本项目客户未及时提供 → 触发异常处理流程（标记"备案缺失"） |

### 四、裁判验证（2名一线投行项目经理反馈）

#### 裁判A（项目经理，5年债券发行经验）

| 反馈项 | 内容 | 严重度 |
|--------|------|--------|
| 角色边界 | "组织者判断重大风险 → 签字决策"这一步，组织者是否具备足够的专业判断能力？建议重大风险判断增加"质控内核+合规审查联席"机制 | 高 |
| 时限刚性 | "产品支持5个工作日设计发行方案"可能不够，复杂项目需要7-10个工作日。建议按项目规模分级设时限（小型3日/中型5日/大型10日） | 中 |
| 异常处理 | "簿记认购不足 → 启动余额包销"需要提前确认公司是否有余额包销授权和资金准备，不能临时决定 | 高 |
| 数据缺失 | "中债登接口未对接"是现实问题，但手工录入容易出错。建议优先对接中债登 | 高 |

#### 裁判B（项目经理，3年债券+2年定增经验）

| 反馈项 | 内容 | 严重度 |
|--------|------|--------|
| 维度设计 | 维度3（市场窗口）中"利率环境"判断依赖10年期国债收益率，但债券发行定价更看同评级信用利差。建议增加"同评级信用利差变化趋势"指标 | 中 |
| 动作衔接 | 事件4（材料准备）和事件5（申报复核）之间，缺少"材料内部预审会"环节。实际工作中，大型项目会先开预审会再正式提交质控 | 中 |
| 合规细节 | 合规审查的"适当性审查"在债券发行中概念不强，债券发行主要是"合规性审查"和"信息披露审查"。建议调整术语 | 低 |
| 跨项目协同 | 同一客户同时有债券和定增项目时，两套模型如何协同？建议增加跨项目冲突检查机制 | 中 |
```

### 6.4 交叉验证的四个原则

| 原则 | 说明 |
|------|------|
| **不自查** | 验证者不能是设计者本人，必须独立视角 |
| **用真案例** | 可执行性验证必须用真实历史项目，不能虚构 |
| **听难听话** | 裁判验证要鼓励挑刺，不能只听表扬 |
| **分优先级** | 问题必须分高/中/低优先级，不能不分轻重 |

## 七、五步法的迭代闭环

```
┌─────────────────────────────────────────┐
│           五步法迭代闭环                 │
├─────────────────────────────────────────┤
│                                          │
│  发布后持续收集数据：                      │
│  ├─ 动作完成率（每个动作按时完成的比例）  │
│  ├─ 超时率（每个动作超时的比例）          │
│  ├─ 异常率（触发异常处理的比例）          │
│  ├─ 人工介入率（需要人工升级的比例）      │
│  └─ 业务人员反馈（评分+文字建议）          │
│                                          │
│  触发迭代条件：                            │
│  ├─ 动作完成率 < 80% → 检查步骤是否过多   │
│  ├─ 超时率 > 20% → 检查时限是否过紧       │
│  ├─ 同一异常 > 10次/周 → 分析异常根因     │
│  ├─ 人工介入率 > 30% → 检查角色设计       │
│  └─ 季度标杆跟踪 → 行业变化 → 模型更新     │
│                                          │
│  迭代方式：                              │
│  ├─ 小迭代：动作微调、时限调整、异常补充   │
│  ├─ 中迭代：维度调整、角色优化、数据新增   │
│  └─ 大迭代：业务目标变化、标杆颠覆、重构   │
│                                          │
└─────────────────────────────────────────┘
```

## 九、快速启动指南

如果你现在要为一个新业务线设计模型，按以下顺序执行：

### 第1天：业务目标（Step 1）
- [ ] 和业务负责人访谈，明确"这条业务线经营什么"
- [ ] 拆解5-7个关键动作
- [ ] 定义每个动作的衡量指标
- [ ] 明确业务边界（和相邻业务线的交界）
- [ ] 输出：《业务目标清单》

### 第2-3天：标杆分析（Step 2）
- [ ] 找3-5个标杆对象（同业+跨业）
- [ ] 收集公开信息（年报/公告/行业报告）
- [ ] 拆解标杆做法（角色/模型/动作/数据/合规）
- [ ] 做差距分析，确定借鉴决策
- [ ] 输出：《标杆分析表》

### 第4-5天：数据关联（Step 3）
- [ ] 拆解每个动作的数据需求
- [ ] 盘点内部数据源（系统/接口/数据库）
- [ ] 盘点外部数据源（公开信息/第三方服务）
- [ ] 评估数据获取路径（自动/半自动/手工）
- [ ] 建立数据关联矩阵
- [ ] 输出：《数据关联矩阵》
- [ ] 检查：自动获取比例是否≥60%？

### 第6-10天：模型定义（Step 4）
- [ ] 定义角色体系（组织者+超级数字员工）
- [ ] 设计业务分析模型（3-5维度）
- [ ] 设计规定动作模型（事件-触发-步骤-输出-异常-时限）
- [ ] 设计模型间衔接
- [ ] 输出：《角色体系》《业务分析模型》《规定动作模型》

### 第11-15天：交叉验证（Step 5）
- [ ] 内部自洽验证（设计团队自查）
- [ ] 标杆对比验证
- [ ] 可执行性验证（1-2个真实案例模拟复盘）
- [ ] 裁判验证（邀请独立业务专家审阅）
- [ ] 汇总问题，分优先级
- [ ] 修正后输出：《交叉验证报告》+ 模型v1.1

### 第16天起：发布与迭代
- [ ] 发布模型v1.0
- [ ] 收集执行数据（完成率/超时率/异常率/人工介入率）
- [ ] 季度标杆跟踪更新
- [ ] 触发迭代条件时启动修正
