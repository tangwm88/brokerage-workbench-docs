# MEMORY.md

## 公司AI转型战略（核心知识）

### 战略目标
**基本目标：** 通过智能体实现业务向前端团队打开  
**理想目标：** 形成公司整体运营管理的自动驾驶

### 架构分层

#### 底座层：公司级自动驾驶大模型
**核心能力：**
- **统一语料**：打通公司全量数据
- **身份打通**：员工账户与系统账户统一
- **关系网络**：关联交易、合同、行为、规则、制度
- **公共能力**：基于AI形成公司级共享能力池

**技术特征：**
- 统一智能体框架
- 智能体可连接组网
- 预判式动作驱动

#### 应用层：八大应用智能体
基于底座二次开发（也可先独立开发后融入）：

1. **资产管理智能体**（产品销售、产品创设服务）
2. **投资银行智能体**（股权、债券、并购业务）
3. **债券销售交易智能体**
4. **机构经纪智能体**
5. **财富管理智能体**
6. **信用交易智能体**
7. **研究所智能体**
8. **固收投资智能体**

### 业务达成模式
- **组织者**：业务支持中心的核心协调角色
- **超级数字员工**：各关键角色的AI分身
- **执行标准**：规定动作 + 动作标准 + 动作模版
- **前端体验**：3-4个关键步骤达成业务或交易

### 设计原则
- **简洁明了**：功能边界清晰，使用门槛低
- **预判驱动**：主动识别需求，提前准备动作
- **提示词工程**：工程级（系统）+ 交互级（用户）双层设计
- **可连接性**：支持智能体之间的协作编排

### 实施路径
1. **能力AI化**：专业能力用AI驱动方式"打开"
2. **工作台化**：向前端提供业务调用的AI工作台
3. **组织扁平化**：打破"割裂的多部门"，走向"统一的能力平台"

### 核心理念
- 不是简单的数字化，而是**组织形态重构**
- 业务支持中心从"被动响应"变为"主动赋能的组织者"
- 前端从"层层请示"变为"直接调用能力"
- 最终形态：公司级自动驾驶式运营管理

## 行业专家知识（经交叉验证）

### 架构设计原则（验证后）
**1. 混合层级架构**
- 底座统一协调 + 八大应用自治
- 符合华泰证券"前端触点智能化、中台能力模块化、后端算法迭代化"实践
- 符合Gartner Multi-Agent Orchestration架构趋势

**2. 智能体通信协议**
- 采用MCP（Model Context Protocol）标准
- 支持A2A（Agent-to-Agent）协议
- 确保智能体间标准化通信

**3. 合规与治理机制**
- 必须包含Human-in-the-loop（关键节点人工确认）
- 符合证券行业监管要求
- 参考沙丘智库金融业智能体最佳实践

### 实施路径（验证后）
```
第一阶段：单智能体试点（3-6个月）
  ↓ 验证业务价值和技术可行性
第二阶段：同业务线多智能体组网（6-12个月）
  ↓ 实现业务线内协作
第三阶段：跨业务线底座统一（12-24个月）
  ↓ 形成公司级自动驾驶能力
```

**行业参考：**
- 红杉AI峰会：单点突破→能力中台→生态进化
- 西南证券：大模型中台+应用层统一API
- 兴业银行：柔性中台体系+平台型组织

### 角色定义（验证后）
| 公司定义 | 行业对应 | 职责说明 |
|----------|----------|----------|
| 组织者 | Agent Orchestrator | 负责任务分解、分配、协调 |
| 超级数字员工 | Role-based Agent | 按角色定义执行规定动作 |
| 人机协作界面 | H-A Interface | Human-Agent交互协议设计 |

**参考框架：**
- CrewAI：role/goal/backstory角色定义法
- 阿里云：Agent Team Lead + 人机协作架构师

### 技术选型参考（验证后）
**1. 提示词工程**
- 工程级（系统提示词）：定义角色、约束、工具
- 交互级（用户提示词）：动态上下文、个性化

**2. 记忆系统**
- 短期记忆：会话上下文保持
- 长期记忆：知识库存储、历史经验

**3. 工具调用**
- 白名单机制：限定可调用的工具范围
- 审批流：高风险操作人工确认

### 关键差异与注意事项
**1. 自主程度（重要差异）**
- 行业趋势：完全自主Agent（主动找活干）
- 公司选择：监督式执行Agent（按规定动作执行）
- **评估**：公司选择更稳妥，符合证券行业强监管要求

**2. 底座定位（细微差异）**
- 行业趋势：智能体引擎/操作系统
- 公司定位：自动驾驶大模型（预判驱动+关系网络）
- **评估**：公司定位更高级，但需分阶段实现

**3. 前端交互（差异较小）**
- 行业主流：自然语言对话
- 公司定位：3-4步极简体验
- **评估**：要求更高，需更强的意图识别能力

### 行业数据参考
- **采用率**：2025年22.2%证券公司采用智能体技术（甲子光年报告）
- **效率提升**：投顾服务响应效率提升80%（证券业协会案例）
- **市场增长**：AI智能体市场复合增长率超40%（2024-2037）

## 证券业务知识库（持续学习中）

### AI化分析框架
**高AI潜力特征：**
- 重复性高、规则明确的流程性任务
- 需要处理大量非结构化数据的分析任务
- 信息整合、内容生成类工作
- 实时监控、预警类任务

**限制因素：**
- 监管合规要求（需人工确认）
- 高风险决策（投资、风控）
- 客户敏感交互（需人工介入）

| 风险预警 | 综合舆情、监管、事件、市场数据；生成客户风险评分；触发分级预警 | 风控规则标准、关系网络 |
| 公司治理跟踪 | 跟踪客户持仓公司的治理结构变化；评估治理风险 | 关系网络、统一语料 |

**角色设计：**
- **组织者：机构经纪协调官**
  - 统筹四大业务目标执行
  - 协调各超级数字员工协作
  - **重点事项请示**：识别需公司层面决策事项，向领导层圆桌会议提请
    - 重点客户支持（战略级客户特殊政策）
    - 重大困难（跨部门协调受阻、资源瓶颈）
    - 重大风险（系统性风险、合规危机）
    - 重要事项（组织架构调整、流程变革）
    - 创新支持（新业务模式、技术突破）
    - 重大投入（超预算资源申请、基础设施升级）
  - **接收战略目标**：接收公司管理层派发的年度/季度战略目标，拆解为可执行任务
  - **考核管理**：执行AI驱动的双重考核机制
- 客户开发员：负责引入客户
- 开户专员：负责账户开立
- 交易服务员：负责交易服务
- 客户关怀员：负责客户服务

**转发边界：**
- 信用交易需求 → 信用交易智能体
- 研究服务需求 → 研究所智能体

**考核机制方案（AI驱动双重考核）：**

| 考核对象 | 考核类型 | 考核内容 | 考核方式 |
|----------|----------|----------|----------|
| 业务支持中心（中后台） | 公司总量考核 | 整体业务目标达成率、服务质量、底座能力贡献度 | AI自动统计：客户转化总数、服务满意度、数据回流质量 |
| 前台业务团队（客户经理） | 单项业务达成考核 | 个人客户引入数、账户开立数、交易服务量、客户服务评分 | AI自动统计：个人业绩看板、客户跟进及时率、业务闭环率 |

**AI驱动考核机制说明：**
- 实时数据采集：底座自动记录每个动作的执行情况和结果
- 智能评分：基于业务规则自动计算考核得分
- 预警提醒：考核指标偏离时自动提醒相关人员
- 双向反馈：前台业绩数据回流到底座，优化客户画像和风控规则

**底座调用：**
- 机构账户关联（查重、画像、持仓查询）
- 风控规则标准（准入审核、交易限额、异常监控）
- 身份权限验证（操作权限范围）

**提示词规范：**
- 使用动词+名词，形容词<10%
- 单句<15字，指令明确
- 覆盖标准业务目标，非具体案例

## 智能体方案评审检查清单（固定评审视角）

### 一、战略契合度评审
| 检查项 | 评审要点 | 通过标准 |
|--------|----------|----------|
| 1.1 底座规范符合性 | 是否基于公司级自动驾驶大模型设计？是否遵循统一语料、身份打通、关系网络规范？ | 明确调用底座能力，不重复建设 |
| 1.2 八大应用定位 | 属于哪个业务线？与现有八大应用的关系？是否重复建设？ | 边界清晰，与现有应用协作而非重复 |
| 1.3 组织转型目标 | 是否体现"割裂的多部门"→"统一能力平台"？是否支持组织扁平化？ | 明确体现组织能力开放和前端赋能 |
| 1.4 业务价值明确 | 解决什么业务痛点？预期ROI？是否可量化？ | 有明确的业务指标和价值衡量标准 |

### 二、架构设计评审
| 检查项 | 评审要点 | 通过标准 |
|--------|----------|----------|
| 2.1 智能体角色定义 | 组织者/超级数字员工角色是否清晰？职责边界是否明确？ | 角色定义完整，符合role/goal/backstory框架 |
| 2.2 组网能力 | 是否支持与其他智能体协作？通信协议是否标准（MCP/A2A）？ | 预留组网接口，遵循标准化协议 |
| 2.3 可连接性 | 输入输出接口是否规范？是否支持被其他智能体调用？ | 接口标准化，文档完整 |
| 2.4 扩展性设计 | 是否支持功能扩展？是否考虑未来业务变化？ | 模块化设计，支持水平扩展 |

### 三、业务逻辑评审
| 检查项 | 评审要点 | 通过标准 |
|--------|----------|----------|

| 3.2 动作模版规范性 | 是否有标准化模版？是否支持快速复制？ | 模版标准化，可复用性强 |
| 3.3 业务流程闭环 | 是否覆盖业务全流程？异常处理机制是否完善？ | 流程完整，异常分支覆盖全面 |
| 3.4 3-4步体验 | 前端达成业务是否只需3-4个关键步骤？ | 体验极简，步骤精简 |

| 检查项 | 评审要点 | 通过标准 |
|--------|----------|----------|

### 五、提示词工程评审
| 检查项 | 评审要点 | 通过标准 |
|--------|----------|----------|
| 5.1 词汇选择 | 是否以动词和名词为主？形容词使用是否 minimal？ | 动词+名词占比>80%，形容词<10% |
| 5.2 表达简洁性 | 句子是否简短？是否避免冗余修饰？ | 单句<15字，无重复表述 |
| 5.3 指令明确性 | 是否为指令性语句？动作是否具体可执行？ | 每句都是明确指令，动作边界清晰 |
| 5.4 通俗易懂 | 是否避免专业术语黑话？业务人员能否理解？ | 无晦涩术语，新人一读即懂 |
| 5.5 结构化分层 | 是否区分系统提示词（工程级）和用户提示词（交互级）？ | 双层结构清晰，职责分离明确 |
| 5.6 版本管理 | 提示词是否有版本控制？变更是否可追溯？ | 有版本号，变更记录完整 |
| 5.7 约束完整性 | 是否明确约束条件？边界是否清晰？ | 包含必须做、禁止做、条件判断 |
| 5.8 输出规范 | 是否定义输出格式？是否提供样例？ | 有标准输出模板，附参考样例 |

### 六、合规与风险评审
| 检查项 | 评审要点 | 通过标准 |
|--------|----------|----------|
| 6.1 Human-in-the-loop | 关键决策点是否必须人工确认？人工介入机制是否完善？ | 关键节点有明确人工确认机制 |
