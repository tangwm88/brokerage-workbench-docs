# 资产管理业务模型蒸馏智能体

> 基于《企业级业务模型蒸馏智能体设计方案 v1.2》+ 资产管理业务知识
> 核心智能体：「资管产品官」+「合规守卫」

## 业务线定位

资产管理业务核心流程：

```
产品创设：需求洞察 → 策略设计 → 结构设计 → 合规审查 → 合同招募 → 备案发行
产品销售：客户画像 → 需求匹配 → 适当性审查 → 签约认购 → 投后服务
```

## 五步法工作流

| 步骤 | 名称 | 输出 | 时限 |
|------|------|------|------|
| Step 1 | 业务目标蒸馏 | 《资管业务目标清单》 | 2日 |
| Step 2 | 标杆分析蒸馏 | 《资管标杆分析表》 | 3日 |
| Step 3 | 数据关联蒸馏 | 《资管数据关联矩阵》 | 3日 |
| Step 4 | 模型定义蒸馏 | 角色体系 + 分析模型 + 动作模型 | 5日 |
| Step 5 | 交叉验证 | 《资管模型验证报告》 | 3日 |

**合计：16个工作日**

## 核心规则（不可修改）

同 model-distillation 技能：陶总五大反对 + 禁用词语表。

参见：`model-distillation` 技能中的核心规则章节。

## 资产管理业务分析模型（输出示例）

### 角色体系：组织者 + 超级数字员工

| 角色 | 本质定位 | 核心职责 | 服务目标 |
|------|---------|---------|---------|
| **资管产品官** | 产品创设与销售支持 | 产品需求洞察、策略回测、结构设计、合同生成、销售匹配 | 资管产品创设 + 产品销售 |
| **合规守卫** | 合规风控专员 | 合规审查、适当性匹配、风险监测、关联交易监控 | 全业务线合规兜底 |
| **组织者** | 整体统筹者 | 统筹产品创设与销售经营，判断风险，调配资源 | 资管业务整体经营 |

### 分析模型：5维度（核心约束：3-5个）

| 维度 | 名称 | 讲人话 | 客观数据 | 输出 |
|------|------|--------|---------|------|
| 1 | **产品需求** | 这个产品有没有市场需求？客户会不会买？ | 市场调研数据/客户需求调研/竞品销售数据 | 需求真实性判断 |
| 2 | **策略质量** | 投资策略能不能赚钱？风险多大？ | 历史回测数据/情景分析/VaR计算 | 策略可行性判断 |
| 3 | **结构合规** | 产品结构是否合规？监管会不会批？ | 监管要求/历史备案通过率/结构参数 | 结构合规性判断 |
| 4 | **客户匹配** | 这个产品适合哪些客户？ | 客户画像/风险测评/投资经验 | 适当性匹配结果 |
| 5 | **投后风险** | 产品运行后有没有风险？ | 净值表现/归因分析/关联交易监测 | 投后风险预警 |

### 规定动作模型：7事件

```
产品立项 → 策略回测 → 结构设计 → 合规审查 → 备案发行 → 产品销售 → 投后服务
   ↑___________________________________________________________↓
                          （发现新需求，循环）
```

| 事件 | 触发条件 | 响应角色 | 风险等级 |
|------|---------|---------|---------|
| 1 产品立项 | 产品需求洞察完成 | 资管产品官 + 组织者 | 必须确认 |
| 2 策略回测 | 产品立项通过 | 资管产品官 | 必须确认 |
| 3 结构设计 | 策略回测通过 | 资管产品官 + 合规守卫 | 必须确认 |
| 4 合规审查 | 结构设计完成 | 合规守卫 | 必须确认 |
| 5 备案发行 | 合规审查通过 | 资管产品官 + 合规守卫 | 必须确认 |
| 6 产品销售 | 产品备案完成 | 资管产品官 | 必须确认 |
| 7 投后服务 | 产品签约完成 | 资管产品官 + 合规守卫 | 可以 |

详细动作设计见 `references/wealth-mgmt-action-model.md`

## 使用方式

### 方式一：对话交互（灵活设计）

当用户说"帮我设计资产管理业务模型"时：

**Step 1：业务目标**
- 问："资管业务经营什么？（产品创设/产品销售/投后服务）"
- 输出：《资管业务目标清单》

**Step 2：标杆分析**
- 问："资管业务的标杆是谁？（华泰/国泰海通/贝莱德/BlackRock）"
- 输出：《资管标杆分析表》

**Step 3：数据关联**
- 问："策略回测需要什么数据？客户画像数据在哪？"
- 输出：《资管数据关联矩阵》

**Step 4：模型定义**
- 输出：角色体系（资管产品官+合规守卫+组织者）+ 分析模型（5维度）+ 动作模型（7事件）

**Step 5：交叉验证**
- 问："找2名资管业务一线人员审阅模型"
- 输出：《资管模型验证报告》

### 方式二：脚本执行（确定性）

```bash
python scripts/distillation_agent.py --input wealth_mgmt.yaml --output ./output
```

## 检查清单

执行五步法前确认：
- [ ] 核心规则已加载（陶总五大反对 + 禁用词语表）
- [ ] 资管业务核心流程已明确（产品创设 + 产品销售 + 投后服务）
- [ ] 高AI适用度环节已识别（策略回测/合同生成/适当性匹配/投后报告）
- [ ] 角色体系已定义（资管产品官 + 合规守卫 + 组织者）
- [ ] 三句话法已融入动作模型（可以/必须确认/不）

## 常见错误

| 错误 | 正确做法 |
|------|---------|
| 把"资管产品"当黑盒 | 拆分为：需求洞察→策略设计→结构设计→合规审查→备案发行 |
| 忽略适当性匹配 | 产品推荐前必须自动评估客户风险承受能力 |
| 投后服务缺失 | 产品卖出后必须持续跟踪净值/风险/关联交易 |
| 策略回测主观化 | 必须基于历史数据，给出VaR/情景分析/归因分析 |
| 合同生成无人工审核 | 合同/招募书AI生成后必须人工逐条审核 |
